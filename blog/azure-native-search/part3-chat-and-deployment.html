Chat History, Streaming, and Docker Deployment

Azure Native Search Series · Part 3 of 3

This is Part 3 of a 3-part series. In Part 1, I covered the architecture. In Part 2, I built the ingestion pipeline. Now let's cover the query pipeline, chat history, streaming, and deployment.

The Query Pipeline: One Call to Rule Them All

When a user asks a question, three things happen:

"How many days per week can employees work remotely?"
     |
     v  Embed question → 1536-dim vector
     |
     v  Azure AI Search (single API call)
     |    +-- BM25 keyword: matches "days", "week", "remotely"
     |    +-- Vector search: HNSW nearest neighbors
     |    +-- Reciprocal Rank Fusion: combine both
     |    +-- Semantic ranker: deep neural reranking
     |
     v  GPT-4o-mini generates cited answer
     |
     v  "Up to 3 days per week [Doc: company_policy.txt, Page: 1]"

Hybrid Search + Semantic Ranking

The retriever is the simplest service in the entire system — 50 lines for a pipeline that replaces 120+ lines in the open-source version:

def hybrid_search(question: str, top_n: int = 5) -> list[dict]:
    client = get_search_client()
    query_embedding = embed_texts([question])[0]

    vector_query = VectorizedQuery(
        vector=query_embedding,
        k_nearest_neighbors=top_n * 4,   # Over-retrieve for better fusion
        fields="embedding",
    )

    try:
        results = client.search(
            search_text=question,
            vector_queries=[vector_query],
            query_type="semantic",
            semantic_configuration_name="default-semantic-config",
            top=top_n,
        )
        candidates = _extract_results(results)
    except Exception:
        # Free tier: fall back to hybrid without semantic ranker
        results = client.search(
            search_text=question,
            vector_queries=[vector_query],
            top=top_n,
        )
        candidates = _extract_results(results)

    return candidates[:top_n]

The k_nearest_neighbors=top_n * 4 is deliberate. We over-retrieve 4x more vector candidates than we need, let the semantic ranker evaluate all of them, then return only the top top_n. More candidates going into the ranker means better final quality.

The try/except handles tier differences gracefully. Azure AI Search Free tier doesn't include the semantic ranker — the code falls back to hybrid (vector + BM25) without it. On Basic+ tier, the semantic ranker kicks in automatically.

Live result:

Question: "How many days per week can employees work remotely?"

Candidate 1: "...remotely up to 3 days per week..."  → score: 3.300
Candidate 2: "...expenses over $25..."                → score: 1.334

The semantic ranker assigns the relevant chunk a score 2.5x higher than the irrelevant one. This is the neural reranking at work — it understands that "3 days per week" answers "how many days" even though the BM25 keyword overlap alone doesn't capture that relationship.

GPT-4o-mini Generation with Citation Enforcement

The generator's system prompt enforces three rules:

SYSTEM_PROMPT = """\
You are a precise document Q&A assistant. Answer the user's question using ONLY \
the retrieved context below. For every factual claim, cite the source as \
[Doc: <filename>, Page: <page>]. If the context does not contain enough \
information to answer, say "I don't have enough information in the indexed \
documents to answer this question." Never fabricate information."""

Three constraints, each preventing a specific failure mode:

  1. "ONLY the retrieved context" — prevents hallucination from parametric knowledge. If the answer isn't in the chunks, the model can't use training data to guess.
  2. "cite the source as [Doc: filename, Page: N]" — forces every claim to have a verifiable reference. Users can check citations against the original document.
  3. "I don't have enough information" — gives the model an explicit escape hatch. Without this, models fabricate answers rather than admitting ignorance.

The context is formatted with numbered source references:

--- Retrieved Context ---
[1] File: company_policy.txt, Page: 1
Acme Corp allows all full-time employees to work remotely up to 3 days per week.
Remote work days must be scheduled in advance...

[2] File: company_policy.txt, Page: 1
Receipts are required for all expenses over $25...
--- End Context ---

User Question: How many days per week can employees work remotely?

Live output: "Employees can work remotely up to 3 days per week [Doc: company_policy.txt, Page: 1]."

Temperature is 0.1 — just enough variation to avoid robotic repetition, low enough to keep answers factual.

Multi-Turn Conversations

async def generate_answer(
    question: str, chunks: list[dict], chat_history: list[dict] | None = None
) -> str:
    messages = [{"role": "system", "content": SYSTEM_PROMPT}]

    if chat_history:
        for msg in chat_history[-6:]:  # Last 3 turns (6 messages)
            messages.append({"role": msg["role"], "content": msg["content"]})

    messages.append({
        "role": "user",
        "content": CONTEXT_TEMPLATE.format(context=context, question=question),
    })

The [-6:] limit keeps the context window manageable — last 3 conversation turns (3 user + 3 assistant messages). This is enough for follow-up questions ("What about for new employees?" after asking about remote work) without bloating the prompt with the entire session.

Chat History: Cosmos DB Design

Project 1 stored chat history in Streamlit's session_state — wiped on every page refresh. Project 2 persists conversations in Azure Cosmos DB, so users can close the browser and resume later.

Schema: Two Document Types, One Container

Database: chatdb
Container: sessions
Partition Key: /session_id

Two document types share the same container:

Session metadata:

{
    "id": "meta_sess_afeb702acc5b",
    "session_id": "sess_afeb702acc5b",
    "title": "How many days per week can employees work remotely?",
    "created_at": "2026-02-19T13:03:23+00:00",
    "updated_at": "2026-02-19T13:03:30+00:00",
    "message_count": 2,
    "type": "session_meta"
}

Messages:

{
    "id": "msg_218f140dd8b7",
    "session_id": "sess_afeb702acc5b",
    "role": "assistant",
    "content": "Employees can work remotely up to 3 days per week [Doc: ...].",
    "sources": [
        {"filename": "company_policy.txt", "page_number": 1,
         "text": "...", "score": 3.300}
    ],
    "created_at": "2026-02-19T13:03:30+00:00",
    "type": "message"
}

Why session_id as Partition Key?

Every query is session-scoped: "get all messages for session X", "add a message to session X", "delete session X". Using session_id as the partition key means:

  • Single-partition reads: ~1 RU per point read (cheapest possible)
  • Co-located data: All messages for a session are physically stored together
  • No hot partitions: A session will never have thousands of messages
  • Efficient deletes: Delete a session = delete all documents in one partition

Cross-partition queries (listing all sessions) cost more but happen rarely — only when loading the session sidebar.

Auto-Titling

Sessions start with the title "New conversation." On the first user message, the session auto-titles itself:

if role == "user" and meta.get("title") == "New conversation":
    meta["title"] = content[:80]

The first 80 characters of the first question become the session title. Simple, no LLM call needed — the user's own words are the best summary.

Session CRUD Endpoints

POST   /chat/sessions                    → Create new session
GET    /chat/sessions                    → List all sessions
GET    /chat/sessions/{id}/messages      → Get messages in session
DELETE /chat/sessions/{id}               → Delete session + all messages

Streaming Responses: Server-Sent Events

The non-streaming endpoint waits for the full answer before responding. For long answers, this creates a noticeable delay. The streaming endpoint sends tokens as they arrive:

@router.post("/stream")
async def query_documents_stream(req: QueryRequest):
    async def event_stream():
        candidates = hybrid_search(req.question, top_n=req.top_k)
        full_answer = []

        async for token in generate_answer_stream(
            req.question, candidates, chat_history
        ):
            full_answer.append(token)
            yield f"data: {token}\n\n"

        # Save complete answer to Cosmos DB
        if req.session_id:
            add_message(req.session_id, "assistant", "".join(full_answer))

        yield "data: [DONE]\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")

The SSE (Server-Sent Events) format is data: <token>\n\n. The client reads tokens as they arrive and appends them to the display. [DONE] signals the end of the stream.

The generator uses Azure OpenAI's native streaming:

stream = await client.chat.completions.create(
    model=settings.azure_openai_chat_deployment,
    messages=messages,
    max_tokens=4096,
    temperature=0.1,
    stream=True,
)

async for chunk in stream:
    if chunk.choices and chunk.choices[0].delta.content:
        yield chunk.choices[0].delta.content

Token-by-token streaming means the user sees the first word within ~500ms, even if the complete answer takes 3–4 seconds to generate.

The Streamlit Frontend

The frontend provides three capabilities: document upload, chat with source citations, and session management.

Chat with Source Citations

Each assistant response includes expandable source citations:

if question := st.chat_input("Ask a question about your documents"):
    with st.chat_message("assistant"):
        with st.spinner("Searching and generating answer..."):
            resp = requests.post(
                f"{API_BASE}/query/",
                json={"question": question, "session_id": session_id, "top_k": 5},
            )
            data = resp.json()
            st.markdown(data["answer"])

            if data["sources"]:
                with st.expander("Sources"):
                    for src in data["sources"]:
                        st.markdown(
                            f"**{src['filename']}** (Page {src['page_number']}) "
                            f"- score: {src['score']:.3f}"
                        )
                        st.caption(src["text"])

Users can expand "Sources" to see the exact chunks the answer was derived from, with retrieval scores and text previews. This transparency builds trust — you can verify that the answer matches the source material.

Session Sidebar

The left sidebar lists all chat sessions with one-click switching. Active sessions are highlighted. Clicking a session loads its full message history from Cosmos DB. Creating a "New Conversation" generates a fresh session_id.

Docker Deployment

Two Containers, No Vector DB

services:
  api:
    build:
      context: .
      dockerfile: Dockerfile.api
    ports:
      - "8000:8000"
    env_file: .env
    healthcheck:
      test: ["CMD", "python", "-c",
             "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"]
      interval: 15s
      timeout: 10s
      retries: 3
      start_period: 30s

  frontend:
    build:
      context: .
      dockerfile: Dockerfile.frontend
    ports:
      - "8501:8501"
    environment:
      - API_BASE=http://api:8000
    depends_on:
      api:
        condition: service_healthy

Compare this to Project 1, which needed a third container for Qdrant with persistent volumes. Azure-native means the vector database is managed — no container, no volumes, no health checks for the vector DB.

The depends_on: condition: service_healthy ensures the frontend doesn't start until the API's health check passes. The API's lifespan handler creates the AI Search index and Cosmos DB on startup, so by the time the health check succeeds, all Azure resources are ready.

Separate Dockerfiles

API (Dockerfile.api):

FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY config.py .
COPY api/ api/
EXPOSE 8000
CMD ["uvicorn", "api.main:app", "--host", "0.0.0.0", "--port", "8000"]

Frontend (Dockerfile.frontend):

FROM python:3.11-slim
WORKDIR /app
RUN pip install --no-cache-dir streamlit requests
COPY frontend/ frontend/
EXPOSE 8501
CMD ["streamlit", "run", "frontend/app.py",
     "--server.port=8501", "--server.address=0.0.0.0"]

The frontend Dockerfile doesn't copy requirements.txt — it only needs streamlit and requests. This keeps the frontend image minimal (~150MB vs ~350MB for the API).

One Command

cp .env.example .env
# Fill in Azure credentials
docker compose up --build

API on port 8000, frontend on port 8501, API docs on port 8000/docs. Total image size: ~500MB (vs ~4GB in Project 1).

Pydantic Settings: Centralized Configuration

class Settings(BaseSettings):
    # Azure OpenAI
    azure_openai_api_key: str
    azure_openai_endpoint: str
    azure_openai_api_version: str = "2024-10-21"
    azure_openai_embedding_deployment: str = "text-embedding-3-small"
    azure_openai_chat_deployment: str = "gpt-4o-mini"

    # Azure AI Search
    azure_search_endpoint: str
    azure_search_api_key: str
    azure_search_index_name: str = "documents"

    # ... 6 more Azure services

    model_config = {"env_file": ".env", "extra": "ignore"}


@lru_cache
def get_settings() -> Settings:
    return Settings()

@lru_cache ensures settings are loaded once and reused. extra = "ignore" means additional environment variables don't cause validation errors. Every service calls get_settings() for its Azure credentials — no hardcoded values anywhere.

Five Lessons Learned

1. Managed services eliminate entire problem categories. I didn't just replace Qdrant with AI Search. I eliminated container health monitoring, persistent volume management, HNSW parameter tuning, manual BM25 scoring, CrossEncoder model downloads, and PyTorch CUDA compatibility. The code is simpler, but the operational overhead reduction is the real win.

2. Tier-aware code handles graceful degradation. The semantic ranker requires Basic+ tier. The retriever tries semantic ranking and falls back to hybrid search on the Free tier. This lets you develop cheaply (Free tier) and deploy with full quality (Basic tier) without code changes.

3. Custom chunking beats framework chunking for simple cases. 99 lines of Python replaced LlamaIndex as a dependency. For sentence-based splitting with overlap, you don't need a framework. The custom chunker is easier to debug, has zero dependencies, and does exactly what you need.

4. Session_id as partition key is the right Cosmos DB pattern. Every access pattern is session-scoped. Partition key = session_id means every query is a cheap single-partition read. This is the canonical Cosmos DB design for chat applications.

5. Plain text parsing should skip Document Intelligence. Sending a .txt file to Document Intelligence wastes a free-tier page and adds 2–3 seconds of latency. The plain text fallback (content.decode("utf-8")) is instant and free. Always optimize for the common case.

The system takes an uploaded document, parses it, chunks it, embeds it, indexes it, and makes it searchable — all through managed Azure services. Questions get answered with citations in 2–4 seconds, conversations persist across sessions, and the whole thing runs in two Docker containers.
