Why I Rebuilt My RAG System on Azure (And Deleted 3,500 Lines of Code)

Azure Native Search Series · Part 1 of 3

This is Part 1 of a 3-part series on building an Azure-native document search system with Azure AI Search, Document Intelligence, OpenAI, Cosmos DB, and Blob Storage. Part 2 covers the ingestion pipeline. Part 3 covers chat history, streaming, and deployment.

The Open-Source RAG Problem

My first project was a standard open-source RAG system: Qdrant for vector search, PyMuPDF for PDF parsing, LlamaIndex for chunking, a CrossEncoder for reranking, and LangChain to glue it together. It worked. It also came with a 4GB Docker image, 14 pip dependencies, a self-hosted vector database container, and ~120 lines just for the retrieval pipeline.

The architecture looked like this:

Project 1 (Open-Source):

  Upload → PyMuPDF parse → LlamaIndex chunk → OpenAI embed → Qdrant upsert

  Question → Qdrant vector search → BM25 rerank (client-side) →
             CrossEncoder rerank (22M params, CPU) → GPT answer

Three separate retrieval steps. A CrossEncoder model running on CPU adding 1.7 seconds of latency. Manual score fusion with hand-tuned weights (0.7 vector / 0.3 BM25). A self-hosted Qdrant container that needed persistent volumes and health checks.

I wanted to know: what happens when you replace every open-source component with its Azure-native equivalent?

The query pipeline went from three retrieval steps and ~120 lines of code to one API call and ~50 lines. The Docker image dropped from ~4GB to ~500MB. The pip dependency count dropped from 14+ (including PyTorch) to 11.

The key insight isn't that Azure is "better" — it's that managed services eliminate entire categories of operational work. No Qdrant health monitoring. No CrossEncoder model downloads. No LlamaIndex version pinning. No PyTorch CUDA compatibility issues.

The Single API Call That Replaces Three Steps

This is the most significant architectural difference. In Project 1, retrieval was:

# Project 1: Three separate steps

# Step 1: Dense vector search (Qdrant)
vector_results = qdrant.search(query_vector, limit=20)

# Step 2: BM25 keyword scoring (rank-bm25 library)
bm25_scores = bm25.get_scores(tokenized_query)

# Step 3: CrossEncoder reranking (22M param model on CPU)
reranked = cross_encoder.predict([(query, doc) for doc in candidates])
# Manual score fusion: 0.7 * vector + 0.3 * bm25

In Project 2, the same retrieval is one call:

# Project 2: One API call does everything

results = client.search(
    search_text=question,                                 # BM25 (server-side)
    vector_queries=[VectorizedQuery(                      # Vector search
        vector=query_embedding,
        k_nearest_neighbors=20,
        fields="embedding",
    )],
    query_type="semantic",                                # Neural reranking
    semantic_configuration_name="default-semantic-config",
    top=5,
)

Azure AI Search handles BM25 scoring, HNSW vector search, Reciprocal Rank Fusion (Microsoft-tuned), and deep neural semantic reranking — all server-side. The client code just sends the query and gets back ranked results with @search.reranker_score.

Why GPT-4o-mini Over a Reasoning Model

Project 1 used gpt-5-mini, a reasoning model. For simple RAG Q&A, this was overkill:

PropertyGPT-4o-minigpt-5-mini
TypeStandard chatReasoning model
TemperatureConfigurableNot supported (fixed)
Hidden reasoningNone~64 tokens of thinking
Cost per 1M tokens$0.15 / $0.60~$3 / $12
Latency~2–4s per answer~8–9s (includes reasoning)

GPT-4o-mini is 20x cheaper and 3x faster for factual retrieval tasks. The reasoning model's hidden thinking adds latency without benefit — for "how many days can employees work remotely?", there's nothing to reason about. The answer is in the context or it isn't.

Why No LangChain

Project 1 used LangChain for prompt construction, chain composition, and LLM calls. Project 2 uses the AsyncAzureOpenAI SDK directly. The generator is 97 lines total — system prompt, context formatting, and the API call.

This wasn't an anti-LangChain decision. For a single-model Q&A system with one prompt template, LangChain adds dependency weight without proportional value. The direct SDK gives you streaming (stream=True), chat history management (just append to messages), and full control over temperature and token limits.

For the multi-agent system (Project 1's sister project with LangGraph), LangChain absolutely earned its place. But for a straightforward RAG pipeline, the direct SDK is simpler.

Live Results: 5/5 Questions Correct

I tested with a sample employee handbook (Acme Corp, 5 chapters, 3,465 bytes). The document was ingested: 1 page parsed, 2 chunks created, 2 vectors indexed. Then five questions:

Every answer included source citations. Every answer was factually correct against the document. The semantic ranker consistently ranked the relevant chunk 2x+ higher than irrelevant ones (scores like 3.30 vs 1.33).
