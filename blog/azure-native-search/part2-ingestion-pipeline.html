Building the Ingestion Pipeline — From Upload to Searchable Index

Azure Native Search Series · Part 2 of 3

This is Part 2 of a 3-part series. In Part 1, I covered the architecture and why Azure-native services replace the open-source stack. Now let's build the ingestion pipeline. Part 3 covers chat history, streaming, and deployment.

The Five-Step Pipeline

User Upload (PDF / DOCX / TXT / Image)
     |
     v  Step 1: Blob Storage
Azure Blob Storage → persistent document store → blob URL
     |
     v  Step 2: Document Intelligence
Azure AI Document Intelligence → structured text with page numbers
     |
     v  Step 3: Custom Chunker
Sentence-based splitting with overlap → list of chunks
     |
     v  Step 4: Embedder
Azure OpenAI text-embedding-3-small → 1536-dim vectors
     |
     v  Step 5: AI Search Index
Azure AI Search → upsert documents with text + vector + metadata
     |
     v
Done → {"pages_parsed": 1, "chunks_created": 2, "chunks_indexed": 2}

One API call from the frontend triggers all five steps. Let me walk through each.

Step 1: Azure Blob Storage — Document Persistence

Before any processing, the raw file is stored in Azure Blob Storage. This ensures the original is always recoverable — if parsing fails, if chunking parameters change, if you need to re-index later.

def upload_document(filename: str, content: bytes, content_type: str) -> str:
    container_client = _ensure_container()
    blob_client = container_client.get_blob_client(filename)
    blob_client.upload_blob(
        content,
        overwrite=True,
        content_settings=ContentSettings(content_type=content_type),
    )
    return blob_client.url

The _ensure_container() call is idempotent — it creates the container on first upload and is a no-op afterwards. overwrite=True means re-uploading a file with the same name replaces it cleanly.

Supported file types: PDF, DOCX, TXT, PNG, JPG, JPEG, TIFF, BMP. The content type is set explicitly for each format so Azure serves the blob correctly.

Cost: Standard LRS storage is $0.018/GB/month. 100MB of documents costs $0.002/month.

Step 2: Azure Document Intelligence — Parsing

This is where Azure shines over PyMuPDF. Document Intelligence's prebuilt-layout model handles:

  • PDFs: Layout-aware extraction with reading order, not just raw text dump
  • DOCX: Structured paragraph and section extraction
  • Images: Full OCR for JPEG, PNG, TIFF, BMP
  • Tables: Structured table extraction (rows, columns, headers)

PyMuPDF extracts raw text from PDFs but misses tables entirely and doesn't support images. For DOCX, you need a separate python-docx library. Document Intelligence handles all formats through one model.

def parse_document(content: bytes, filename: str) -> list[ParsedPage]:
    client = get_doc_intelligence_client()

    poller = client.begin_analyze_document(
        model_id="prebuilt-layout",
        analyze_request=content,
        content_type="application/octet-stream",
    )
    result = poller.result()

    pages: list[ParsedPage] = []
    if result.pages:
        for page in result.pages:
            page_text_lines = []
            if page.lines:
                for line in page.lines:
                    page_text_lines.append(line.content)

            text = "\n".join(page_text_lines).strip()
            if text:
                pages.append(ParsedPage(
                    text=text,
                    page_number=page.page_number,
                    filename=filename,
                ))

    # Fallback: if page-level extraction fails, use full content
    if not pages and result.content:
        pages.append(ParsedPage(text=result.content, page_number=1, filename=filename))

    return pages

The begin_analyze_document() call is asynchronous on Azure's side — it returns a poller that you call .result() on. For a 5-page PDF, this typically takes 2–5 seconds. For a 100-page document, 15–30 seconds.

Plain Text Optimization

For .txt files, we bypass Document Intelligence entirely. There's no layout to analyze — just decode the bytes to UTF-8:

def parse_plain_text(content: bytes, filename: str) -> list[ParsedPage]:
    text = content.decode("utf-8", errors="replace").strip()
    if not text:
        return []
    return [ParsedPage(text=text, page_number=1, filename=filename)]

This saves both time and cost. The free tier gives 500 pages/month — no reason to waste a page on a text file.

Step 3: Custom Sentence Chunker — Why I Wrote My Own

Project 1 used LlamaIndex's SentenceWindowNodeParser. It works well, but it pulls in llama-index-core and its entire dependency tree. For a sentence-based chunker, that's heavy.

The custom chunker is 99 lines. It splits on sentence boundaries (.!? followed by whitespace) and maintains configurable overlap:

def chunk_pages(pages: list[ParsedPage]) -> list[Chunk]:
    settings = get_settings()
    max_chars = settings.chunk_size * 4      # ~4 chars per token
    overlap_chars = settings.chunk_overlap * 4

    chunks: list[Chunk] = []
    global_index = 0

    for page in pages:
        sentences = _split_into_sentences(page.text)
        current_chunk_sentences: list[str] = []
        current_length = 0

        for sentence in sentences:
            sentence_len = len(sentence)

            if current_length + sentence_len > max_chars and current_chunk_sentences:
                # Emit current chunk
                chunk_text = " ".join(current_chunk_sentences)
                chunks.append(Chunk(
                    text=chunk_text,
                    filename=page.filename,
                    page_number=page.page_number,
                    chunk_index=global_index,
                ))
                global_index += 1

                # Keep overlap sentences for next chunk
                overlap_sentences: list[str] = []
                overlap_len = 0
                for s in reversed(current_chunk_sentences):
                    if overlap_len + len(s) > overlap_chars:
                        break
                    overlap_sentences.insert(0, s)
                    overlap_len += len(s)

                current_chunk_sentences = overlap_sentences
                current_length = overlap_len

            current_chunk_sentences.append(sentence)
            current_length += sentence_len

        # Emit final chunk
        if current_chunk_sentences:
            chunk_text = " ".join(current_chunk_sentences)
            chunks.append(Chunk(text=chunk_text, ...))
            global_index += 1

    return chunks

Why Sentence Boundaries?

Splitting mid-sentence creates chunks where the first or last statement is incomplete. The model can't properly understand "...optimizing computational efficiency and" or "implementing robust data privacy measures..." without the surrounding context. Sentence-boundary splitting ensures every chunk contains complete thoughts.

Why Overlap?

Without overlap, a fact that spans two chunks ("The policy was updated in January. New rates take effect in March.") gets split across boundaries. With 128-token overlap (~512 characters), the last few sentences of chunk N are duplicated at the start of chunk N+1. This redundancy ensures cross-boundary facts appear in at least one complete chunk.

Configuration

chunk_size: int = 512       # ~2,048 characters per chunk
chunk_overlap: int = 128    # ~512 characters of overlap

The 512-token target balances two concerns: too small (128 tokens) means each chunk lacks context and retrieval returns fragments. Too large (2048 tokens) means too much noise per chunk, wasting the LLM context window. 512 tokens is a sweet spot for English prose — roughly 2–3 paragraphs per chunk.

Step 4: Azure OpenAI Embedding

Each chunk is converted to a 1536-dimensional vector using text-embedding-3-small:

def embed_texts(texts: list[str]) -> list[list[float]]:
    settings = get_settings()
    client = get_openai_client()
    response = client.embeddings.create(
        input=texts,
        model=settings.azure_openai_embedding_deployment,
    )
    return [item.embedding for item in response.data]

The entire function is 7 lines. Batch embedding is built into the API — you pass a list of texts and get back a list of vectors. For large documents with hundreds of chunks, the ingest route processes in batches of 100.

Why text-embedding-3-small Over text-embedding-3-large?

ModelDimensionsCost per 1M tokens
text-embedding-3-small1,536$0.02
text-embedding-3-large3,072$0.13

The small model is 6.5x cheaper with half the vector dimensions. For document-level search (not fine-grained semantic similarity), the quality difference is negligible. And halving dimensions halves the AI Search storage cost.

Step 5: Azure AI Search Index — Schema and Upsert

The index schema defines what's searchable and how:

fields = [
    SimpleField(name="id", type=String, key=True),
    SearchableField(name="text", type=String, analyzer_name="en.microsoft"),
    SearchField(name="embedding", type=Collection(Single),
                searchable=True, vector_search_dimensions=1536,
                vector_search_profile_name="default-vector-profile"),
    SimpleField(name="filename", type=String, filterable=True, facetable=True),
    SimpleField(name="page_number", type=Int32, filterable=True, sortable=True),
    SimpleField(name="chunk_index", type=Int32, sortable=True),
    SimpleField(name="created_at", type=String),
]

Each field serves a specific retrieval purpose:

FieldTypePurpose
idString (key)UUID per chunk
textSearchableBM25 keyword search + semantic ranker input
embeddingVector (1536)HNSW dense search
filenameFilterableFilter results by source document
page_numberFilterable + SortableCitation metadata + page filtering
chunk_indexSortableReconstruct document order
created_atStringAudit trail

Vector Search Configuration

vector_search = VectorSearch(
    algorithms=[HnswAlgorithmConfiguration(name="default-hnsw")],
    profiles=[VectorSearchProfile(
        name="default-vector-profile",
        algorithm_configuration_name="default-hnsw",
    )],
)

HNSW (Hierarchical Navigable Small World) is the vector search algorithm. It builds a graph structure over the vectors for fast approximate nearest-neighbor search. Azure AI Search manages the HNSW index automatically — no manual tuning of ef_construction or M parameters needed.

Semantic Search Configuration

semantic_config = SemanticConfiguration(
    name="default-semantic-config",
    prioritized_fields=SemanticPrioritizedFields(
        content_fields=[SemanticField(field_name="text")],
    ),
)

This tells the semantic ranker which field contains the main content for deep neural reranking. When you query with query_type="semantic", Azure runs its cross-encoder model on the text field of the top candidates — similar to what CrossEncoder does locally in Project 1, but server-side with a Microsoft-trained model.

Index Creation: Idempotent

def ensure_index() -> None:
    client = get_index_client()
    index = SearchIndex(
        name=settings.azure_search_index_name,
        fields=fields,
        vector_search=vector_search,
        semantic_search=semantic_search,
    )
    client.create_or_update_index(index)

create_or_update_index() is idempotent — it creates the index on first call and updates the schema on subsequent calls. This runs at API startup via the lifespan handler, so the index is always ready before the first request.

Document Upsert

def upsert_chunks(chunks: list[Chunk], embeddings: list[list[float]]) -> int:
    documents = [
        {
            "id": str(uuid.uuid4()),
            "text": chunk.text,
            "embedding": embedding,
            "filename": chunk.filename,
            "page_number": chunk.page_number,
            "chunk_index": chunk.chunk_index,
            "created_at": datetime.now(timezone.utc).isoformat(),
        }
        for chunk, embedding in zip(chunks, embeddings)
    ]

    batch_size = 1000
    total = 0
    for i in range(0, len(documents), batch_size):
        batch = documents[i : i + batch_size]
        result = client.upload_documents(documents=batch)
        total += sum(1 for r in result if r.succeeded)

    return total

Batching at 1,000 documents handles large files efficiently. The succeeded check counts only successfully indexed chunks — any failures (malformed data, quota exceeded) are silently excluded from the total.

Why Manual Push Instead of AI Search Auto-Indexing?

Azure AI Search can automatically pull from Blob Storage using built-in indexers. I chose manual push for three reasons:

  1. Full pipeline visibility — I can see exactly what was parsed, how it was chunked, and how many vectors were created
  2. Custom chunking — AI Search's built-in chunking doesn't support sentence-boundary awareness
  3. Debugging — when retrieval quality is poor, I can inspect each pipeline step independently

The Complete Ingest Route

All five steps are orchestrated by a single FastAPI endpoint:

@router.post("/", response_model=IngestResponse)
async def ingest_document(file: UploadFile = File(...)):
    suffix = Path(file.filename).suffix.lower()
    if suffix not in ALLOWED_EXTENSIONS:
        raise HTTPException(status_code=400, detail=f"Unsupported type '{suffix}'")

    content = await file.read()

    # Step 1: Blob Storage
    blob_url = upload_document(file.filename, content, content_type)

    # Step 2: Document Intelligence (or plain text fallback)
    if suffix == ".txt":
        pages = parse_plain_text(content, file.filename)
    else:
        pages = parse_document(content, file.filename)

    # Step 3: Chunk
    chunks = chunk_pages(pages)

    # Step 4: Embed (in batches of 100)
    texts = [chunk.text for chunk in chunks]
    embeddings = []
    for i in range(0, len(texts), 100):
        embeddings.extend(embed_texts(texts[i : i + 100]))

    # Step 5: Index
    ensure_index()
    indexed = upsert_chunks(chunks, embeddings)

    return IngestResponse(
        filename=file.filename, blob_url=blob_url,
        pages_parsed=len(pages), chunks_created=len(chunks),
        chunks_indexed=indexed,
    )

For a 5-page PDF, this typically takes 5–10 seconds end-to-end. For a plain text file, under 3 seconds (no Document Intelligence call).
