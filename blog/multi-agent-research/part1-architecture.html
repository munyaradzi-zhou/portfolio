Why I Built a Multi-Agent Research System (And Why Single Agents Fail)

Multi-Agent Research Series · Part 1 of 3

This is Part 1 of a 3-part series on building a production multi-agent research system with LangGraph, OpenAI GPT-4o, Qdrant, and Tavily Search. Part 2 covers building the agents. Part 3 covers deployment and live results.

The Problem with Single-Agent Research

Ask GPT-4o to "research the challenges of deploying LLMs in production and write a professional report." You'll get something reasonable — maybe 400 words of general observations, no citations, no data points, and no way to know if the claims are accurate.

That's the single-agent problem. One prompt has to handle research, analysis, writing, AND quality control simultaneously. It's like asking one person to be the reporter, data analyst, editor, and fact-checker all at once. The result is mediocre at everything.

Single Agent (Traditional):

  Question → One LLM prompt → Answer

  Problems:
  - No real web search (uses training data only)
  - No structured data extraction
  - No citation verification
  - No quality feedback loop
  - No revision mechanism

I wanted something fundamentally different: a system where specialized agents collaborate in a pipeline, each focused on one task, with a built-in quality gate that can send work back for revision.

The Multi-Agent Architecture

The system uses four specialized agents orchestrated by LangGraph's StateGraph:

User Query
    |
    v
+-------------+
|  Researcher  |  ← Tavily Search (3 parallel queries, 23+ sources)
|              |  ← Qdrant Memory (recalls prior research)
+------+------+
       v
+-------------+
|   Analyst    |  ← Extracts 7 themes, 5+ data points, finds gaps
+------+------+
       v
+-------------+
|   Writer     |  ← 2+ citations per section, 1+ statistic per section
+------+------+
       v
+-------------+
|   Critic     |  ← Scores 0-10 on accuracy, completeness, clarity
|              |  ← Routes back to Writer if any score < 7
+------+------+
       |
  score ≥ 7 → Final Report (approved)
  score < 7 → Back to Writer (max 3 revisions)

Each agent has its own system prompt, its own temperature setting, and its own tools. They share state through a single TypedDict, and LangGraph handles the orchestration.

Why Four Agents?

The number isn't arbitrary. Each agent maps to a distinct cognitive task with different requirements:

AgentTaskTemperatureWhy Separate?
ResearcherWeb search + synthesis0.1Needs tool access (Tavily, Qdrant). Low temperature for factual accuracy.
AnalystJSON extraction0.0Pure structured output. Zero creativity needed.
WriterMarkdown report0.3Slight creativity for readability. Uses Analyst's structure.
CriticScore + route0.0Objective evaluation. Must be deterministic.

The key insight: the Analyst and Writer have fundamentally incompatible requirements. The Analyst needs zero-temperature deterministic JSON extraction. The Writer needs some creative flexibility for readability. Combining them in one prompt forces a compromise that weakens both.

Similarly, the Critic must be independent from the Writer. You can't ask an agent to evaluate its own work objectively — the same way a newspaper editor doesn't write the articles they review.

Why LangGraph?

I evaluated three frameworks: LangChain agents (ReAct loop), CrewAI, and LangGraph.

LangChain ReAct agents are great for single-agent tool use but don't support structured multi-agent pipelines. There's no concept of "agent A's output flows to agent B."

CrewAI is purpose-built for multi-agent workflows, but it abstracts away the graph structure. When I needed conditional edges (Critic → Writer if score < 7, else → End), CrewAI's sequential/hierarchical modes didn't map cleanly.

LangGraph gives you a directed graph where nodes are functions and edges define the flow:

class AgentState(TypedDict):
    task: str              # Original query
    research: str          # Raw research notes
    analysis: dict         # Structured JSON from Analyst
    draft: str             # Current report draft
    critique: str          # Critic's JSON evaluation
    revision_count: int    # Number of revision cycles
    final_output: str      # Approved final report

def build_graph() -> CompiledStateGraph:
    sg = StateGraph(AgentState)
    sg.add_node("researcher", researcher_node)
    sg.add_node("analyst", analyst_node)
    sg.add_node("writer", writer_node)
    sg.add_node("critic", critic_node)

    sg.add_edge(START, "researcher")
    sg.add_edge("researcher", "analyst")
    sg.add_edge("analyst", "writer")
    sg.add_edge("writer", "critic")
    sg.add_conditional_edges("critic", should_revise,
                             {"writer": "writer", "end": END})
    return sg.compile()

That's the entire orchestration layer. Six nodes, five edges, one conditional. The simplicity is the point — the complexity lives in the agents themselves, not in the plumbing.

The Shared State Pattern

Every agent reads from and writes to the same AgentState dictionary. This eliminates the need for message passing between agents — the state IS the message.

AgentState flow:

  Researcher writes → state["research"] = "5,375 chars of notes..."
  Analyst reads state["research"], writes → state["analysis"] = {7 themes, 5 data points}
  Writer reads state["analysis"], writes → state["draft"] = "6,399 chars of report..."
  Critic reads state["draft"], writes → state["critique"] = {scores, status, feedback}

When the Critic sends work back to the Writer, it doesn't need a separate message — the Writer reads state["critique"] and state["revision_count"] to understand what to fix. The revision count also acts as a safety valve: after 3 cycles, the Critic force-approves to prevent infinite loops.

The system produced a 589-word professional report with inline citations in every section, 5 quantitative data points, and an 8.7/10 quality score — all without a single revision cycle. That's the multi-agent advantage: each specialist produces high-quality output in its domain, so the final product passes quality review on the first try.
