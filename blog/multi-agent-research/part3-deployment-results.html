The API Layer: REST + WebSocket

# Compile the LangGraph once at startup
@asynccontextmanager
async def lifespan(app: FastAPI):
    global graph
    graph = build_graph()
    yield

# Synchronous — waits for full pipeline
@app.post("/research")
async def run_research(request: ResearchRequest):
    result = await graph.ainvoke(initial_state)
    return ResearchResponse(
        final_report=result["final_output"],
        revision_count=result["revision_count"],
        critique_summary=result["critique"],
    )

# Real-time streaming — sends updates as each agent completes
@app.websocket("/ws/research")
async def research_stream(websocket: WebSocket):
    await websocket.accept()
    async for chunk in graph.astream(initial_state):
        for agent_name, agent_output in chunk.items():
            await websocket.send_json({
                "agent": agent_name,
                "status": "completed",
                "content": json.dumps(agent_output),
            })
    await websocket.send_json({
        "agent": "system", "status": "done",
        "content": "Research pipeline completed",
    })

The key design decision: graph.ainvoke() vs graph.astream(). Both execute the same pipeline. ainvoke() returns only the final state — perfect for REST. astream() yields incremental updates as each node completes — perfect for WebSocket that powers the real-time frontend.

WebSocket Integration in Streamlit

Streamlit's execution model is synchronous — every widget interaction re-runs the entire script. The solution: run the WebSocket client in a background thread and poll for updates.

def _run_websocket_pipeline(query: str, result_holder: dict):
    """Runs in a background thread."""
    import websockets.sync.client as ws_sync

    with ws_sync.connect(WS_URL, close_timeout=300) as ws:
        ws.send(json.dumps({"query": query}))
        while True:
            raw = ws.recv()
            data = json.loads(raw)
            result_holder["updates"].append(data)
            if data["status"] in ("done", "error"):
                break

# In the main Streamlit thread:
thread = threading.Thread(target=_run_websocket_pipeline,
                          args=(query, result_holder), daemon=True)
thread.start()

# Poll for updates every 0.5s
while not result_holder["done"]:

      - qdrant_data:/qdrant/storage

depends_on ensures Qdrant starts before the API, and restart: unless-stopped keeps both services running after crashes. First-time startup takes ~30 seconds. Subsequent starts are under 5 seconds thanks to Docker layer caching.

Seven Lessons Learned

1. Calibrate your Critic, or it will destroy your pipeline. The single biggest improvement from v1.0 to v2.0 was adding calibration rules to the Critic's scoring rubric. Without them, the Critic scored Completeness at 5–6/10 on every report, triggering 3 revision cycles every time. Adding "3+ angles with citations → auto ≥ 7" eliminated this problem overnight.

2. Enforce quantitative constraints in prompts. Telling an LLM to "extract data points" produces 0–2 data points. Telling it "extract a minimum of 5 quantitative data points including market sizes, growth rates, dates, and performance metrics" produces exactly 5.

3. Parallel search is free performance. Running 3 Tavily searches with asyncio.gather() takes the same time as 1 search but gives you 4.6x the sources.

4. Max 2 revision instructions prevents analysis paralysis. When the Critic returns 5 revision instructions, the Writer tries to address all of them and does a poor job on each. Capping at 2 focused instructions produces much better targeted revisions.

5. Graceful degradation beats hard failures. If Qdrant is down, the Researcher skips memory recall and does fresh web searches. The pipeline never crashes — it just runs slightly less informed.

6. WebSocket streaming changes the user experience. A 108-second pipeline feels like "the app is broken" with a spinner. With WebSocket streaming showing each agent's progress, the same 108 seconds feels like watching a team work.

7. The hardest part isn't the code — it's the prompts. The entire orchestration layer is 63 lines. The API is 173 lines. The prompts template is 154 lines — and it's the file that changed the most between v1.0 and v2.0. The code is plumbing; the prompts are the product.
