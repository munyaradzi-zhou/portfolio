Building the Agents — From Web Search to Self-Critiquing Reports

The Tools Layer

Before diving into agents, let's look at the three tools they share.

Tavily Search — Structured Web Results

Most web search APIs return raw HTML or snippets you have to parse. Tavily returns structured JSON with relevance scores:

async def tavily_search(query: str, max_results: int = 5) -> list[dict]:
    client = AsyncTavilyClient(api_key=os.getenv("TAVILY_API_KEY"))
    response = await client.search(query, max_results=max_results,
                                    search_depth="advanced")
    return [
        {"title": r["title"], "url": r["url"],
         "content": r["content"], "score": r["score"]}
        for r in response.get("results", [])
    ]

Each result comes with a relevance score (0.0–1.0). The search_depth="advanced" setting enables deep crawling, pulling content from within pages rather than just metadata.

Why Tavily over Serper/SerpAPI? Structured output (no HTML parsing), built-in relevance scoring, and async support. The free tier gives 1,000 searches/month.

Qdrant Memory — Cross-Session Knowledge

The Researcher doesn't start from zero on every query. Qdrant stores research notes from prior sessions, so the system accumulates institutional knowledge over time.

class MemoryStore:
    def __init__(self):
        self.client = AsyncQdrantClient(url=os.getenv("QDRANT_URL"))
        self.openai = AsyncOpenAI()

    async def store(self, content: str, metadata: dict) -> str:
        embedding = await self._embed(content[:2000])
        point_id = str(uuid4())
        await self.client.upsert("agent_memory", points=[
            PointStruct(id=point_id, vector=embedding,
                       payload={"content": content[:2000], **metadata})
        ])
        return point_id

    async def recall(self, query: str, top_k: int = 3) -> list[dict]:
        embedding = await self._embed(query)
        results = await self.client.search("agent_memory",
                                            query_vector=embedding, limit=top_k)
        return [{"content": r.payload["content"],
                 "score": r.score, **r.payload} for r in results]

The embedding model is text-embedding-3-small (1,536 dimensions, cosine distance). The 2,000-character limit on stored content is deliberate — we want the key findings, not the raw dump.

Graceful degradation: Every Qdrant call is wrapped in try/except. If the vector database is down, the pipeline continues without memory. No crashes, no retries, just a slightly less informed first step.

Agent 1: The Researcher

The Researcher is the most complex agent. Its job: turn a vague query into 5,000+ characters of structured, cited research notes.

Multi-Pass Search Strategy

Rather than sending the user's raw query to Tavily once, the Researcher generates three targeted sub-queries and searches in parallel:

async def researcher_node(state: AgentState) -> dict:
    task = state["task"]
    memory = MemoryStore()

    # Step 1: Check long-term memory for prior research
    prior = await memory.recall(task, top_k=3)

    # Step 2: Generate 3 targeted sub-queries (GPT-4o-mini)
    sub_queries = await _generate_sub_queries(task)

    # Step 3: Run 3 Tavily searches IN PARALLEL (8 results each)
    search_tasks = [tavily_search(q, max_results=8) for q in sub_queries]
    all_results = await asyncio.gather(*search_tasks)

    # Step 4: Flatten and deduplicate by URL
    seen_urls = set()
    unique_results = []
    for results in all_results:
        for r in results:
            if r["url"] not in seen_urls:
                seen_urls.add(r["url"])
                unique_results.append(r)
    # 3 x 8 = 24 results → ~23 after dedup

    # Step 5: Synthesize into structured notes (GPT-4o)
    research_notes = await _synthesize(task, unique_results, prior)

    # Step 6: Store in memory for future sessions
    await memory.store(research_notes, {"task": task, "type": "research"})

    return {"research": research_notes}

Why 3 sub-queries? A single search gets you ~5 results from one angle. Three targeted searches from different angles yield ~23 unique sources after URL deduplication — a 4.6x amplification in source coverage with essentially the same wall-clock time, thanks to asyncio.gather().

Timing breakdown: Memory recall: 1.51s. Sub-query generation (GPT-4o-mini): ~2.0s. 3x Tavily search (parallel): ~8.0s. GPT-4o synthesis: ~24.0s. Memory store: ~2.0s. Total: 38.93s.

Agent 2: The Analyst

The Analyst converts 5,375 characters of unstructured research notes into a precise JSON structure. Temperature: 0.0 — zero creativity, maximum determinism.

Enforced Data Extraction

The system prompt doesn't just ask for "key insights." It demands specific structures:

{
  "key_themes": ["theme1", "theme2", ...],
  "main_findings": [
    {
      "finding": "Deploying LLMs requires substantial GPU resources",
      "evidence": "GPT-4 has up to a trillion parameters",
      "source": "https://seldon.io/...",
      "importance": "high"
    }
  ],
  "data_points": [
    {
      "metric": "Number of parameters in GPT-4",
      "value": "up to a trillion",
      "source": "https://seldon.io/..."
    }
  ],
  "gaps": ["Missing cost data for deployment strategies"],
  "recommended_structure": ["Introduction", "Section 1", ...]
}

The critical constraint: data_points must contain 5+ entries. This forces the model to dig for quantitative information rather than making vague qualitative statements.

Live output: 7 themes, 5 findings (3 high-importance, 2 medium), 5 data points, 2 identified gaps, and a recommended 7-section report structure. Timing: 19.30 seconds.

Agent 3: The Writer

Citation and Statistics Enforcement

The system prompt includes explicit requirements:

For EVERY section you write:
  1. Include at least 2 inline citations [Source: URL]
  2. Include at least 1 specific number, statistic, or date
  3. Pair every challenge with a concrete solution
  4. Use the Analyst's recommended_structure as your outline

Before submitting, verify:
  [ ] Every section has 2+ citations?
  [ ] Every section has a statistic?
  [ ] Any vague claims without numbers?
  [ ] Executive Summary includes 2-3 key numbers?

This checklist approach works because GPT-4o treats explicit enumerated requirements as hard constraints. Without these rules, the Writer produces generic paragraphs. With them, every section contains verifiable claims with sources.

Revision Awareness

async def writer_node(state: AgentState) -> dict:
    analysis = state["analysis"]
    revision_count = state.get("revision_count", 0)
    critique = state.get("critique", "")

    user_msg = f"Analysis:\n{json.dumps(analysis, indent=2)}\n\n"
    user_msg += f"Task: {state['task']}\n"

    if revision_count > 0 and critique:
        user_msg += f"\n--- REVISION #{revision_count} ---\n"
        user_msg += f"Previous critique:\n{critique}\n"
        user_msg += "Address the specific issues above.\n"

Live output: 6,399 characters, 589 words, 10+ inline cited sources. Every section contains at least 2 citations and 1 data point. Timing: 32.61 seconds.

Agent 4: The Critic

The Critic is the quality gate — the component that transforms this from "a pipeline that generates text" into "a system that guarantees quality."

Calibrated Scoring Rubrics

ACCURACY (0-10):
  9-10: Every claim has a citation, no unsupported assertions
  7-8:  Most claims cited, minor gaps
  5-6:  Multiple unsupported claims

COMPLETENESS (0-10):
  9-10: Covers all major angles with depth
  7-8:  Covers main angles, some deeper exploration needed
  5-6:  Major aspects missing
  Calibration: 3+ angles with citations → auto ≥ 7

CLARITY (0-10):
  9-10: Excellent structure, logical flow
  7-8:  Good structure, minor readability issues
  5-6:  Disorganized or hard to follow
  Calibration: Exec Summary + body + Takeaways → auto ≥ 7

The calibration notes are critical. In v1.0, the Critic was over-harsh — scoring Completeness at 5–6/10 because the report didn't cover every tangential subtopic. This triggered unnecessary revision loops. The calibration rules establish baseline expectations and prevent the LLM's tendency toward perfectionism from creating infinite loops.

The Routing Decision

def should_revise(state: AgentState) -> str:
    critique = json.loads(state["critique"])
    if critique["status"] == "APPROVED":
        return "end"
    if state["revision_count"] >= 3:
        return "end"  # Safety valve
    return "writer"

The max-2-instructions limit on revision feedback is deliberate. Giving the Writer a laundry list of 5+ issues leads to poor revisions — the model tries to address everything and addresses nothing well. Two focused instructions produce targeted improvements.

Live evaluation: Accuracy 9/10, Completeness 8/10, Clarity 9/10 — 8.7/10 APPROVED on the first pass. No revisions needed. Timing: 16.84 seconds.

The v1.0 → v2.0 Evolution

The most important lesson: prompt engineering is system design, not just writing words.

Componentv1.0 Problemv2.0 Fix
ResearcherSingle search query → 5 results3 parallel sub-queries → 23+ unique sources (4.6x)
AnalystNo minimum data point requirementEnforced 5+ data points
WriterNo citation enforcement → generic paragraphs2+ citations per section mandate
CriticUncalibrated rubrics → endless revision loopsCalibrated rubrics with auto-floor rules

The total prompt changes were maybe 200 lines across 4 templates. But those 200 lines transformed a system that looped 3x on every query into one that approves on the first pass with an 8.7/10 score.
