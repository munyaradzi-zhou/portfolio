The API Layer: FastAPI with Two Delivery Modes

REST Endpoint: Simple but Slow

@app.post("/research", response_model=ResearchResponse)
async def run_research(request: ResearchRequest):
    initial_state = {
        "task": request.query,
        "research": "", "analysis": {}, "draft": "",
        "critique": "", "revision_count": 0, "final_output": "",
    }
    result = await graph.ainvoke(initial_state)

    return ResearchResponse(
        task=request.query,
        final_report=result.get("final_output", result.get("draft", "")),
        revision_count=result.get("revision_count", 0),
        critique_summary=result.get("critique"),
    )

The response comes back after ~97 seconds with the full report. Simple, but the user stares at a loading spinner for over a minute.

WebSocket Endpoint: Real-Time Agent Progress

LangGraph's graph.astream() yields each agent's output as it completes. We pipe these directly to a WebSocket connection:

@app.websocket("/ws/research")
async def research_stream(websocket: WebSocket):
    await websocket.accept()
    data = await websocket.receive_json()
    query = data.get("query", "")

    initial_state = {
        "task": query,
        "research": "", "analysis": {}, "draft": "",
        "critique": "", "revision_count": 0, "final_output": "",
    }

    async for chunk in graph.astream(initial_state):
        for agent_name, agent_output in chunk.items():
            await websocket.send_json({
                "agent": agent_name,
                "status": "completed",
                "content": json.dumps(agent_output, default=str),
            })

    await websocket.send_json({
        "agent": "system",
        "status": "done",
        "content": "Financial analysis pipeline completed",
    })

The client sees 9 progressive updates instead of one long wait:

The Streamlit Frontend

The frontend is a Streamlit app with live streaming — the WebSocket connection pipes agent updates directly to the UI.

Execution Modes

WebSocket (Real-time): Shows a live progress bar as each agent completes. The user sees which agent is currently working, how long each step took, and the revision cycle in real time.

REST (Synchronous): Simpler fallback that waits for the full result. Useful for environments where WebSocket connections are unreliable.

Docker: One Container Instead of Two

The original multi-agent-research system required two Docker containers: one for the API and one for Qdrant. By migrating to Azure Cosmos DB, the financial analysis system needs only one:

services:
  api:
    build: .
    ports:
      - "8001:8001"
    env_file:
      - .env
    command: uvicorn api.main:app --host 0.0.0.0 --port 8001

No Qdrant container, no volume mounts, no inter-container networking. The memory store lives in Azure Cosmos DB, so the container is stateless — you can scale it horizontally, restart it freely, and deploy it anywhere that can reach Azure's APIs.

Seven Lessons Learned

1. The Critic Needs Careful Calibration — Or It Ruins Everything. My first Critic prompt said "Score the report (0–10)." Every report got 6/10 and entered an infinite revision loop. Adding explicit calibration anchors ("proper disclaimer + metrics + risk = at least 7") fixed this overnight.

2. Parallel Search is a Force Multiplier. One search query returns 5–8 results from one angle. Three targeted queries running in parallel return 22 unique results from three angles in roughly the same wall-clock time.

3. Client-Side Vector Search Has Limits — And That's OK. Computing cosine similarity over 9 documents takes 1.36 seconds. Fine for now. At 10,000 documents, switch to Cosmos DB's vector indexing or a dedicated vector database.

4. Temperature Tuning Per Agent Matters. My initial version used 0.3 for everything. The Analyst's JSON output was inconsistent. Setting Analyst and Critic to 0.0 made their output deterministic. The Writer at 0.3 writes naturally; at 0.7, it gets creative with financial claims (bad).

5. "Preserve What's Good" Is the Key Revision Instruction. Without explicit guidance, the Writer rewrites from scratch on each revision, losing improvements. Adding "Make targeted improvements while preserving the parts that were already good" stabilized the revision process.

6. Azure-Native Simplifies Everything. One Azure subscription, one bill, one set of credentials, one Docker container. The trade-off is vendor lock-in — but migrating from Azure OpenAI back to OpenAI direct is a one-line change.

7. The Revision Loop Adds Real Value — Measurably. The approved report (Draft 3) was 39% longer than Draft 1, with specific competitor strategies replacing generic statements, and causal risk chains replacing surface-level observations. The 0.25-point score increase doesn't capture the qualitative improvement.
