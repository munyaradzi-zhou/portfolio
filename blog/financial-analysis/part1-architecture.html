Why I Built a Multi-Agent System for Financial Analysis (And Why Single Agents Fail)

Multi-Agent Financial Analysis Series · Part 1 of 3

This is Part 1 of a 3-part series on building a multi-agent financial market analysis system with LangGraph, Azure OpenAI, and Azure Cosmos DB. Part 2 covers building the agents. Part 3 covers deployment and live results.

The Problem: One LLM Prompt Can't Do Everything

I started where most people start: a single prompt.

"You are a financial analyst. Research Apple's stock and write an investment report."

The result? A confident-sounding report with hallucinated P/E ratios, no source citations, missing disclaimers, and a "Strong Buy" recommendation based on information the model made up. The LLM was trying to simultaneously be a researcher, a data analyst, a writer, and a quality reviewer — and failing at all four.

Financial analysis is uniquely unforgiving. A wrong revenue figure isn't just a bad answer — it's the kind of mistake that erodes trust in the entire output. And unlike a general research task where vague answers might pass, investment analysis demands:

  • Specific numbers with source attribution (P/E ratio of 33.46x from Yahoo Finance, not "around 30x")
  • Mandatory disclaimers ("This does not constitute investment advice...")
  • Balanced analysis (bull AND bear case, not just cheerleading)
  • Risk disclosure (macroeconomic, competitive, regulatory)

No single prompt can reliably deliver all of these. The cognitive tasks are fundamentally different — web searching requires breadth, data extraction requires precision, report writing requires creativity, and quality review requires critical thinking. Asking one LLM call to excel at all four is like asking one employee to be the researcher, accountant, copywriter, and compliance officer simultaneously.

The Solution: Four Agents, One Pipeline

The insight that unlocked the project was simple: separate the cognitive tasks into specialised agents, each with its own prompt, tools, and focus area.

Each agent does one thing well:

AgentSpecialityTemperatureWhy This Temperature
ResearcherGathers market data, earnings reports, analyst opinions from the web0.1Mostly factual, slight variation in phrasing
AnalystExtracts structured JSON — P/E ratios, revenue, growth rates, risk factors0.0Deterministic extraction, no creativity needed
WriterProduces a professional investment analysis report with disclaimers0.3Creative writing with professional tone variation
CriticScores on accuracy, completeness, risk disclosure, clarity (0–10)0.0Consistent, repeatable evaluation

The magic is in the revision loop. If the Critic scores the report below 7 on any dimension, it sends specific feedback back to the Writer. The Writer revises, the Critic re-evaluates, and this continues for up to 3 cycles.

In our live test, the Apple stock analysis went through exactly 3 revisions:

  • Draft 1: 5,412 chars → Critic: 8.25/10, REVISE
  • Draft 2: 5,931 chars → Critic: 8.25/10, REVISE
  • Draft 3: 7,547 chars → Critic: 8.50/10, APPROVED

The report grew by 39% through self-critique, adding a key metrics table, expanded competitive analysis, and more detailed macroeconomic risk scenarios.

Why LangGraph (Not CrewAI, Not AutoGen, Not Raw LangChain)

LangChain (raw chains): Linear only. A → B → C → D. No way to loop the Critic back to the Writer. Dead end.

CrewAI: High-level abstraction that's great for quick prototypes but hides too much. I needed fine-grained control over the state passed between agents, the conditional routing logic, and the streaming behavior.

AutoGen: Designed for conversational multi-agent chat, not pipeline orchestration. The agents "talk to each other" which is elegant for brainstorming but adds unpredictable latency and token cost for a structured pipeline.

LangGraph: Stateful directed graph with conditional edges. Exactly what I needed.

# The entire pipeline definition — supervisor.py
graph = StateGraph(AgentState)

graph.add_node("researcher", researcher_node)
graph.add_node("analyst", analyst_node)
graph.add_node("writer", writer_node)
graph.add_node("critic", critic_node)

graph.set_entry_point("researcher")
graph.add_edge("researcher", "analyst")
graph.add_edge("analyst", "writer")
graph.add_edge("writer", "critic")

# This is the key — conditional routing
graph.add_conditional_edges(
    "critic",
    should_revise,  # Returns "writer" or "end"
    {"writer": "writer", "end": END},
)

LangGraph gave me three things that mattered:

  1. Conditional edges — The Critic-to-Writer revision loop, cleanly expressed
  2. Shared state — All agents read and write a common AgentState TypedDict (no message passing, no serialization overhead)
  3. Streaming — graph.astream() yields each agent's output as it completes, perfect for WebSocket streaming to the frontend

The Shared State: AgentState

class AgentState(TypedDict):
    task: str              # "Analyze Apple's stock performance..."
    research: str          # 3,493 chars of structured research notes
    analysis: dict         # JSON with key_metrics, market_trends, risk_factors
    draft: str             # 5,412–7,547 chars of markdown report
    critique: str          # JSON scoring with revision instructions
    revision_count: int    # 0, 1, 2, or 3
    final_output: str      # Set by Critic on APPROVED

Each agent reads the fields it needs and writes exactly one or two fields back. No agent knows about the others. No agent calls another agent. They just read state, do their work, and write state. LangGraph handles the routing.

GPT-4o-mini is not GPT-4o — it's smaller and faster, but for this use case, the quality difference is negligible. The Critic catches quality issues. The 16x cost reduction (from ~$1.00 per run to ~$0.07) makes the system practical for repeated use.

Qdrant → Azure Cosmos DB

Qdrant is a purpose-built vector database with HNSW indexing. But it requires a separate container — and for a financial analysis system with a small memory store (< 100 documents), that's overkill.

Azure Cosmos DB (Serverless tier) gave me zero ops, pay-per-use (~286 RUs per memory store), and one Azure subscription for everything. The trade-off: no native vector search at this tier, so I compute cosine similarity client-side.

# Embed the query
query_vector = await self._embed(query)  # text-embedding-3-small

# Fetch all documents from Cosmos DB
items = list(container.query_items(
    query="SELECT c.id, c.content, c.embedding FROM c WHERE c.embedding != null",
    enable_cross_partition_query=True,
))

# Compute cosine similarity client-side
scored = [(cosine_similarity(query_vector, item["embedding"]), item) for item in items]
scored.sort(key=lambda x: x[0], reverse=True)
return scored[:top_k]

For 9 documents, this takes 1.36 seconds. Not as fast as Qdrant's native HNSW, but perfectly adequate. If the memory store grows to thousands of documents, I'd switch to Cosmos DB's vector indexing feature or a dedicated vector database.
