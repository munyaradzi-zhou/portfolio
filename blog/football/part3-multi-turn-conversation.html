Building a Football Agentic RAG System: Multi-Turn Conversation and Context Tracking (Part 3)

Football Analytics Series · Part 3 of 3

This is Part 3 of a 3-part series on building a Football Agentic RAG System. Part 1 covered the multi-agent architecture and temporal knowledge graph design. Part 2 covered hybrid retrieval and provenance tracking. This post covers multi-turn conversation and context tracking.

Introduction

Natural conversations about football rarely consist of isolated questions. A user might ask "What are Manchester United's stats this season?" and then follow up with "Who is their top scorer?" and "What about last season?" Each follow-up question contains pronouns and implicit references that rely on context from previous turns.

Traditional RAG systems treat each query independently, forcing users to repeat information: "Who is Manchester United's top scorer?" and "What were Manchester United's stats last season?" This is tedious and unnatural. What we need is a multi-turn conversation system that tracks context across messages and enriches follow-up questions with missing entities.

This final post in the series dives into the conversation system: how the SessionService tracks message history and context entities, how the OrchestratorAgent enriches queries to resolve pronouns and implicit references, and concrete examples of multi-turn conversations about teams, players, and competitions spanning dozens of turns.

Multi-Turn Conversation Requirements

A robust multi-turn conversation system for football analytics must handle:

1. Pronoun Resolution

Users naturally use pronouns to refer to previously mentioned entities:

  • "Show me Messi's stats." → "What about his World Cup performance?"
  • "How did Manchester United perform?" → "They finished 8th in the Premier League."
  • "Who won the Champions League?" → "What were their semifinal matches?"

The system must resolve "his", "they", and "their" to the correct entities (Messi, Manchester United, Champions League winner).

2. Implicit Entity References

Users often omit entities that were mentioned in previous turns:

  • "Manchester United's stats" → "Who is their top scorer?" (implicit: Manchester United)
  • "Messi's goals" → "What about last season?" (implicit: Messi)
  • "Champions League 2023" → "Show me the final" (implicit: Champions League 2023)

The system must track which team, player, competition, or season was last discussed and inject that context into follow-up queries.

3. Context Switching

Users may switch topics mid-conversation:

  • "Manchester United's stats" → "Now show me Barcelona" → "What about their performance?" (now refers to Barcelona, not Manchester United)

The system must update context when new entities are introduced and use the most recent entity for ambiguous references.

4. Temporal Context

Users often refer to temporal periods discussed earlier:

  • "Stats for 2023-2024" → "What about the previous season?" (implicit: relative to 2023-2024)

The system must track which seasons, dates, or time ranges were mentioned and resolve relative references accordingly.

SessionService: Message History and Context Tracking

The SessionService is the core component responsible for managing conversation state. It stores message history, extracts context entities from messages, and provides context to the orchestrator for query enrichment.

Session Schema

Each session is stored in Redis with the following structure:

{
  "session_id": "football_abc123",
  "messages": [
    {
      "role": "user",
      "content": "What are Manchester United's stats?",
      "timestamp": "2024-01-15T10:30:00",
      "metadata": {}
    },
    {
      "role": "assistant",
      "content": "Manchester United finished 8th in the Premier League...",
      "timestamp": "2024-01-15T10:30:05",
      "metadata": {
        "confidence": 0.95,
        "sources_count": 3,
        "trace_id": "xyz789"
      }
    }
  ],
  "context_teams": ["manchester united"],
  "context_players": [],
  "context_competitions": [],
  "context_seasons": ["2024"],
  "created_at": "2024-01-15T10:30:00"
}

The session tracks:

  • messages: Full conversation history (user and assistant messages)
  • context_teams: List of teams mentioned in the conversation
  • context_players: List of players mentioned
  • context_competitions: List of competitions mentioned
  • context_seasons: List of seasons or years mentioned

Each list maintains the order of mentions, with the most recent entity at the end. This enables the system to use the last mentioned entity for ambiguous references.

Adding Messages to the Session

The add_message method stores a message and extracts entities from it:

class SessionService:
    def add_message(
        self, session_id: str, role: str, content: str,
        metadata: Optional[Dict[str, Any]] = None
    ) -> None:
        """Add a message to the conversation history."""
        session = self.get_session(session_id) or self._initialize_session()

        message = {
            "role": role,
            "content": content,
            "timestamp": datetime.utcnow().isoformat(),
            "metadata": metadata or {}
        }

        session["messages"].append(message)

        # Extract and update context entities from user messages
        if role == "user":
            self._update_context_entities(session, content, metadata)

        # Keep only last 20 messages to prevent memory bloat
        if len(session["messages"]) > 20:
            session["messages"] = session["messages"][-20:]

        self.create_session(session_id, session)

Context Entity Extraction

The _update_context_entities method uses pattern matching to detect teams, players, competitions, and seasons:

def _update_context_entities(
    self, session: Dict[str, Any], message: str,
    metadata: Optional[Dict[str, Any]]
) -> None:
    """Extract and update context entities from user message."""
    message_lower = message.lower()

    # Common team patterns
    teams = [
        "manchester united", "real madrid", "barcelona", "liverpool",
        "bayern munich", "chelsea", "arsenal", "manchester city"
    ]
    for team in teams:
        if team in message_lower and team not in session.get("context_teams", []):
            session.setdefault("context_teams", []).append(team)

    # Common player patterns
    players = ["messi", "ronaldo", "neymar", "mbappe", "haaland"]
    for player in players:
        if player in message_lower and player not in session.get("context_players", []):
            session.setdefault("context_players", []).append(player)

    # Keep only last 5 of each entity type
    for key in ["context_teams", "context_players", "context_competitions", "context_seasons"]:
        if key in session and len(session[key]) > 5:
            session[key] = session[key][-5:]

Query Enrichment for Follow-Up Questions

The OrchestratorAgent enriches queries using the _enrich_with_context method:

def _enrich_with_context(
    self, query: str, message_history: list = None,
    context_entities: dict = None
) -> str:
    """Enrich follow-up questions with context from conversation history."""
    if not context_entities:
        return query

    query_lower = query.lower()
    enriched = query

    # Check for pronouns or implicit references
    needs_enrichment = any(word in query_lower for word in [
        "they", "them", "their", "he", "him", "his",
        "that team", "the team", "last season"
    ])

    if not needs_enrichment:
        return query

    # Add team context if available
    teams = context_entities.get("teams", [])
    if teams and any(ref in query_lower for ref in ["the team", "they", "them"]):
        enriched = f"{teams[-1].title()} - {enriched}"

    # Add player context if available
    players = context_entities.get("players", [])
    if players and any(ref in query_lower for ref in ["he", "him", "his"]):
        enriched = f"{players[-1].title()} - {enriched}"

    return enriched

[Extract: players=["messi"]]

User: What about his World Cup performance?

Scenario 3: Context Switching

Turn 1
User: What are Manchester United's stats?
[Extract: teams=["manchester united"]]
Assistant: Manchester United stats...

Turn 2
User: Now show me Barcelona
[Extract: teams=["manchester united", "barcelona"]]
Assistant: Barcelona stats...

Turn 3
User: What about their performance?
[Context: teams=["manchester united", "barcelona"]]
[Most recent: "barcelona"]
[Enriched: "Barcelona - What about their performance?"]

When Barcelona is introduced, it becomes the most recent team. The query "their performance" now refers to Barcelona, not Manchester United.

Implementation Benefits

The multi-turn conversation system provides several key benefits:

Natural interaction. Users can ask follow-up questions using pronouns and implicit references, just like talking to a human analyst. No need to repeat entity names in every query.

Context persistence. The session tracks up to 20 messages and 5 entities per type. This covers most conversation lengths while preventing unbounded memory growth.

Explicit enrichment logging. Every query enrichment is logged, making it easy to debug when context resolution fails. The logs show "Context enrichment: 'Who is their top scorer?' → 'Manchester United - Who is their top scorer?'"

Graceful fallback. If no context entities are available, queries pass through unchanged. The system does not hallucinate entities that were never mentioned.

Future Enhancements

Several improvements could further enhance the conversation system:

NER-based entity extraction. Replace substring matching with spaCy or similar NER models to detect entities not in the curated lists. This would handle player nicknames, team abbreviations, and less common entities.

Coreference resolution. Use a coreference resolution model to properly resolve "he", "they", and "it" to specific entities based on linguistic context, not just recency.

Intent classification for context switching. Detect when the user explicitly wants to switch context ("Now show me Barcelona") vs continue the current topic ("What about them?").

Conversation summarization. After 20+ turns, summarize the conversation history to maintain context without exceeding token limits when passing history to the LLM.

Conclusion

The Football Agentic RAG System demonstrates how multi-agent architecture, temporal knowledge graphs, hybrid retrieval, provenance tracking, and multi-turn conversation can be combined to create a powerful football analytics platform. The system handles complex temporal reasoning ("last 5 seasons"), multi-hop graph traversal ("players who scored against their former clubs"), explanatory questions grounded in match reports, and natural follow-up questions across dozens of conversation turns.

The three-part architecture (specialized agents, hybrid retrieval, conversation tracking) provides modularity, extensibility, and factual accuracy. Each component can be improved independently: swap the entity extraction method, add a new agent, or upgrade the knowledge graph schema without rewriting the entire system.

This architecture is directly applicable to other domains requiring temporal reasoning and multi-modal retrieval: financial analytics ("What was Apple's revenue growth in Q4 2023?" → "How does that compare to last year?"), scientific literature review ("Show me papers on graph neural networks from 2023" → "What about their applications in drug discovery?"), or legal document analysis ("Find cases citing Brown v. Board of Education" → "What were the main arguments?").

The key insight is that complex reasoning tasks benefit from decomposition into specialized agents, each optimized for a specific retrieval or reasoning pattern, coordinated by an orchestrator that maintains conversation state and ensures factual grounding through provenance tracking.
