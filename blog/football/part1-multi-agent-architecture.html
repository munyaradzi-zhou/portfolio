Building a Football Agentic RAG System: Multi-Agent Architecture and Temporal Knowledge Graphs (Part 1)

Football Analytics Series · Part 1 of 3

This is Part 1 of a 3-part series on building a Football Agentic RAG System that combines temporal knowledge graphs, multi-agent orchestration, and retrieval-augmented generation for football analytics. Part 2 covers hybrid retrieval and provenance tracking. Part 3 covers multi-turn conversation and context tracking.

Introduction

Imagine asking a football analytics system: "How many goals did Harry Kane score in the last 5 seasons?" This seemingly simple question requires complex reasoning across multiple dimensions:

  • Temporal normalization: What does "last 5 seasons" mean? Which years? (2019-2020, 2020-2021, 2021-2022, 2022-2023, 2023-2024)
  • Entity recognition: Who is "Harry Kane"? What is his player ID in the database?
  • Query routing: Should this query go to the knowledge graph, SQL database, or vector store?
  • Data aggregation: Sum goals across all competitions, all clubs, all matches in those seasons.
  • Answer synthesis: Format the result into a natural language response with context.

Traditional RAG systems struggle with this multi-step reasoning. Vector similarity search alone cannot reliably traverse temporal relationships, aggregate statistics, or maintain factual precision across 32 seasons of football data. What we need is a multi-agent system where specialized agents handle different aspects of the reasoning pipeline, coordinated by an orchestrator that understands the structure of football knowledge.

This project builds exactly that: a Football Agentic RAG System powered by temporal knowledge graphs in Neo4j, hybrid retrieval across graph, SQL, and vector stores, and a multi-agent architecture inspired by OpenAI's Temporal Agents pattern. The system can answer complex football questions spanning decades of data, provide explanations grounded in match reports, track multi-turn conversations, and prevent hallucinations through rigorous provenance tracking.

Why Multi-Agent Architecture for Football Analytics?

Football data is inherently multi-modal and requires different reasoning strategies depending on the question type:

Question Type Example Best Data Source Reasoning Required
Statistical "How many goals did Messi score?" SQL (aggregation) Sum, count, average
Relational "When did Barcelona beat Real Madrid?" Neo4j (graph traversal) Pattern matching, relationships
Explanatory "Why did the World Cup final go to extra time?" Vector store (RAG over match reports) Semantic search, summarization
Temporal "Show stats for the last 5 seasons" All sources + temporal filter Date range normalization

A monolithic RAG pipeline cannot efficiently handle all four question types. Statistical questions need aggregation. Relational questions need graph traversal. Explanatory questions need vector similarity. Temporal questions need date range normalization across all sources.

The solution is to decompose the system into specialized agents, each responsible for a specific aspect of football reasoning:

  1. Orchestrator Agent: Coordinates the workflow, manages context, enriches queries
  2. Temporal Normalizer Agent: Parses temporal expressions ("last season", "2023/24", "previous 5 seasons") into structured filters
  3. Query Router Agent: Determines which data source(s) to query based on question intent
  4. Knowledge Graph Query Agent: Generates and executes Cypher queries against Neo4j
  5. Stats Aggregation Agent: Generates and executes SQL queries against PostgreSQL
  6. Explanation RAG Agent: Performs semantic search over match reports and articles
  7. Visualization Agent: Generates chart specifications for data visualization

Each agent operates independently but communicates through the orchestrator, which maintains conversation state, tracks provenance, and synthesizes the final response.

Temporal Knowledge Graph Design

The core of the system is a temporal knowledge graph built in Neo4j that models 32 seasons of football data (1992-2024). The schema captures the rich relational structure of football: teams play matches, players score goals, competitions have seasons, managers lead clubs.

Schema Overview

The Neo4j schema contains the following node types:

// Core Entity Nodes
(:Team {name, country, founded, stadium})
(:Player {name, nationality, position, birth_date})
(:Match {date, season, home_score, away_score, competition})
(:Competition {name, type, country})
(:Season {year, start_date, end_date})
(:Manager {name, nationality})
(:Stadium {name, city, capacity})

// Event Nodes (for granular match events)
(:Goal {minute, type, player_id, match_id})
(:Card {minute, type, player_id, match_id})
(:Substitution {minute, player_in, player_out, match_id})

And the following relationship types:

// Match Relationships
(Team)-[:PLAYED_HOME]->(Match)
(Team)-[:PLAYED_AWAY]->(Match)
(Team)-[:WON]->(Match)
(Team)-[:LOST]->(Match)
(Team)-[:DREW]->(Match)

// Player Relationships
(Player)-[:SCORED]->(Goal)
(Player)-[:ASSISTED]->(Goal)
(Player)-[:RECEIVED]->(Card)
(Player)-[:PLAYED_FOR {season, from_date, to_date}]->(Team)

// Competition Relationships
(Match)-[:PART_OF]->(Competition)
(Competition)-[:HAS_SEASON]->(Season)

// Manager Relationships
(Manager)-[:MANAGED {from_date, to_date}]->(Team)

Design Rationale

Temporal properties on relationships. The PLAYED_FOR and MANAGED relationships include temporal properties (season, from_date, to_date). This allows us to query "Which teams did Ronaldo play for in 2008?" by filtering on the relationship's temporal properties, not just the nodes. This is critical for handling transfers, loan periods, and managerial changes.

Explicit match results as relationships. Rather than computing match results on the fly, we materialize WON, LOST, and DREW relationships during ingestion. This makes queries like "Show all matches where Barcelona won" trivial graph traversals rather than score comparisons.

Event nodes for granularity. Goals, cards, and substitutions are modeled as separate nodes rather than properties of the match. This enables queries like "Show all goals scored in the 90th minute or later" and "Which players have scored against their former clubs?" without complex property filtering.

Bidirectional match relationships. Each match has both PLAYED_HOME and PLAYED_AWAY relationships. This supports symmetric queries: "Show all home matches for Arsenal" and "Show all away matches against Liverpool" are equally efficient.

Example: Multi-Hop Query

Consider the question: "Which players scored when Barcelona beat Real Madrid in El Clasico during the 2022-2023 season?"

This requires a 4-hop graph traversal:

MATCH (barcelona:Team {name: "FC Barcelona"})-[:WON]->(match:Match)
      <-[:PLAYED_AWAY]-(real_madrid:Team {name: "Real Madrid"}),
      (match)-[:PART_OF]->(competition:Competition {name: "La Liga"}),
      (player:Player)-[:SCORED]->(goal:Goal)-[:SCORED_IN]->(match)
WHERE match.season = "2022-2023"
RETURN player.name, goal.minute
ORDER BY goal.minute

This query traverses relationships from team to match, match to competition, match to goal, and goal to player, filtering on temporal properties (season) and match result (Barcelona won). A vector-only RAG system would struggle to reliably extract this chain of connections from text chunks.

Multi-Agent Orchestration

The OrchestratorAgent is the central coordinator that manages the entire query processing pipeline. It receives the user's question, enriches it with conversation context, delegates to specialized agents, and synthesizes the final response.

Orchestrator Workflow

Here is the core process_query method from orchestrator_agent.py:

class OrchestratorAgent:
    def __init__(self):
        self.temporal_agent = TemporalNormalizerAgent()
        self.router_agent = QueryRouterAgent()
        self.kg_agent = KGQueryAgent()
        self.stats_agent = StatsAggregationAgent()
        self.rag_agent = ExplanationRAGAgent()
        self.viz_agent = VisualizationAgent()

    async def process_query(
        self,
        query: str,
        session_id: str = None,
        trace_id: str = None,
        message_history: list = None,
        context_entities: dict = None
    ) -> Dict[str, Any]:
        """
        Main entry point for processing user queries.
        """
        logger.info(f"Orchestrator received query: {query}")

        # Track provenance
        if trace_id:
            provenance_service.add_step(trace_id, "orchestrator_started", {
                "query": query,
                "has_history": bool(message_history),
                "context_entities": context_entities or {}
            })

        # 1. Enrich query with context for follow-up questions
        enriched_query = self._enrich_with_context(
            query, message_history, context_entities
        )

        # 2. Normalize temporal expressions
        temporal_filters = self.temporal_agent.normalize(enriched_query)
        context = {
            "filters": temporal_filters,
            "session_id": session_id,
            "trace_id": trace_id,
            "message_history": message_history,
            "context_entities": context_entities
        }

        # 3. Route query to determine primary data source
        route_plan = self.router_agent.route(enriched_query)
        primary_source = route_plan.get("primary")

        # 4. Execute appropriate sub-agent
        result = {}
        if primary_source == "neo4j":
            result = self.kg_agent.process_query(enriched_query, context)
        elif primary_source == "sql":
            result = self.stats_agent.process_query(enriched_query, context)
        elif primary_source == "vector_store":
            result = self.rag_agent.process_query(enriched_query, context)

        # 5. Generate visualization if data is available
        viz_spec = None
        if result.get("data"):
            viz_spec = self.viz_agent.generate_chart_spec(
                result["data"], enriched_query
            )

        # 6. Construct final response
        return {
            "answer": result.get("answer"),
            "sources": result.get("sources", []),
            "visualization": viz_spec,
            "data": result.get("data"),
            "debug_info": {
                "route": route_plan,
                "temporal_filters": temporal_filters
            }
        }

The workflow follows a clear six-step pipeline:

  1. Context enrichment: Add entities from conversation history to handle follow-up questions
  2. Temporal normalization: Parse "last 5 seasons" into ["2019-2020", "2020-2021", ...]
  3. Query routing: Determine which data source is best suited for this question
  4. Agent execution: Delegate to the appropriate specialized agent
  5. Visualization generation: Create chart specifications if data is available
  6. Response synthesis: Combine answer, sources, visualization, and debug info

Query Enrichment for Multi-Turn Conversations

The _enrich_with_context method handles follow-up questions by adding previously mentioned entities to the query. This enables natural conversations like:

User: "What are Manchester United's stats this season?"
Assistant: "Manchester United finished 8th in the Premier League 2023-24..."

[Context: teams=["manchester united"]]

User: "Who is their top scorer?"
[Enriched: "Manchester United - Who is their top scorer?"]
Assistant: "Bruno Fernandes was their top scorer with 15 goals..."

Here is the enrichment logic:

def _enrich_with_context(
    self, query: str, message_history: list = None,
    context_entities: dict = None
) -> str:
    """
    Enrich follow-up questions with context from conversation history.
    """
    if not context_entities:
        return query

    query_lower = query.lower()
    enriched = query

    # Check for pronouns or implicit references
    needs_enrichment = any(word in query_lower for word in [
        "they", "them", "their", "he", "him", "his",
        "that team", "the team", "last season"
    ])

    if not needs_enrichment:
        return query

    # Add team context if available
    teams = context_entities.get("teams", [])
    if teams and not any(team in query_lower for team in teams):
        last_team = teams[-1]
        if any(ref in query_lower for ref in ["the team", "they", "them"]):
            enriched = f"{last_team.title()} - {enriched}"

    # Add player context if available
    players = context_entities.get("players", [])
    if players and not any(player in query_lower for player in players):
        last_player = players[-1]
        if any(ref in query_lower for ref in ["he", "him", "his"]):
            enriched = f"{last_player.title()} - {enriched}"

    logger.info(f"Context enrichment: '{query}' → '{enriched}'")
    return enriched

The enrichment checks for pronouns and implicit references, then prepends the most recently mentioned team, player, or competition to disambiguate the query. This simple heuristic dramatically improves multi-turn conversation quality.

Agent Specialization

Each specialized agent implements a specific reasoning pattern optimized for its data source.

Temporal Normalizer Agent

The TemporalNormalizerAgent parses natural language temporal expressions into structured filters that other agents can use. It supports both LLM-based parsing and rule-based fallback.

class TemporalNormalizerAgent:
    def normalize(self, query: str) -> Dict[str, Any]:
        """
        Parse temporal expressions from the query.
        Returns structured filters like {'season': '2023-2024'}.
        """
        # Try LLM-based normalization first
        if self.llm_service:
            try:
                filters = self._normalize_with_llm(query)
                if filters:
                    return filters
            except Exception as e:
                logger.debug(f"LLM failed, falling back to rules: {e}")

        # Fallback to rule-based normalization
        return self._normalize_with_rules(query)

    def _normalize_with_rules(self, query: str) -> Dict[str, Any]:
        """Rule-based temporal normalization."""
        filters = {}
        query_lower = query.lower()

        # Season pattern: "2023/24", "2023-24", "2023-2024"
        season_pattern = r"(\d{4})[/-](\d{2,4})"
        match = re.search(season_pattern, query)
        if match:
            year1 = match.group(1)
            year2 = match.group(2)
            if len(year2) == 2:
                year2 = "20" + year2
            filters["season"] = f"{year1}-{year2}"

        # "Last season" logic
        if "last season" in query_lower:
            filters["season"] = "2023-2024"

        # "Previous 5 seasons"
        if "last 5 seasons" in query_lower:
            filters["seasons"] = [
                "2019-2020", "2020-2021", "2021-2022",
                "2022-2023", "2023-2024"
            ]

        return filters

The agent tries LLM-based parsing first (for complex expressions like "during Mourinho's first season at Chelsea") but falls back to regex patterns for common cases. This hybrid approach ensures robustness.

Query Router Agent

The QueryRouterAgent analyzes the question to determine the optimal data source(s). It classifies questions into intent categories and routes accordingly:

class QueryRouterAgent:
    def route(self, query: str) -> Dict[str, Any]:
        """
        Determine the best data source(s) for this query.
        Returns: {'primary': 'sql', 'secondary': ['neo4j'], 'intent': 'statistics'}
        """
        query_lower = query.lower()
        plan = {"primary": None, "secondary": [], "intent": "unknown"}

        # 1. Explanatory questions (WHY/HOW) -> Vector Store
        if "why" in query_lower or "how" in query_lower:
            plan["primary"] = "vector_store"
            plan["secondary"] = ["neo4j", "sql"]
            plan["intent"] = "explanation"
            return plan

        # 2. Statistical questions (HOW MANY, AVERAGE, TOP) -> SQL
        if any(x in query_lower for x in [
            "how many", "average", "total", "most", "top", "stats"
        ]):
            plan["primary"] = "sql"
            plan["secondary"] = ["neo4j"]
            plan["intent"] = "statistics"
            return plan

        # 3. Relationship questions (WHO PLAYED, TRANSFER, HISTORY) -> Neo4j
        if any(x in query_lower for x in [
            "played for", "transfer", "history", "beat", "won against"
        ]):
            plan["primary"] = "neo4j"
            plan["secondary"] = ["sql"]
            plan["intent"] = "relationship"
            return plan

        # Default fallback
        plan["primary"] = "vector_store"
        plan["intent"] = "general"
        return plan

The routing logic uses keyword matching to classify intent. More sophisticated implementations could use an LLM classifier or learned model, but simple keyword matching achieves high accuracy for football queries.

Knowledge Graph Query Agent

The KGQueryAgent generates and executes Cypher queries against Neo4j. It uses the LLM to translate natural language into Cypher, leveraging few-shot examples from the schema:

class KGQueryAgent:
    def process_query(
        self, query: str, context: Dict[str, Any] = None
    ) -> Dict[str, Any]:
        """Process a query using the Knowledge Graph."""
        # Get schema information
        schemas = self.schema_tool.get_schemas()
        neo4j_schema = schemas["neo4j_schema"]

        # Generate Cypher query using LLM
        cypher_query = self.llm_service.generate_cypher(
            query, neo4j_schema, context
        )

        # Execute the query
        results = self.neo4j_tool.execute_query(cypher_query)

        # Format the answer
        if results:
            answer = self._format_answer(query, results)
            return {
                "answer": answer,
                "data": results,
                "sources": [{"type": "neo4j", "query": cypher_query}],
                "source": "neo4j"
            }
        else:
            return {
                "answer": "No matches found in the knowledge graph.",
                "data": [],
                "sources": [],
                "source": "neo4j"
            }

The agent relies on the LLM service to generate syntactically correct Cypher queries grounded in the schema. It includes the temporal filters from context to ensure queries respect season boundaries.

Why This Architecture Works

The multi-agent architecture provides several critical advantages over monolithic RAG systems:

Separation of concerns. Each agent is responsible for a single aspect of reasoning. The temporal agent does not need to know about Cypher generation. The KG agent does not need to know about SQL. This modularity makes the system easier to maintain, test, and extend.

Hybrid retrieval. Different question types require different data sources. Statistical questions benefit from SQL aggregation. Relational questions benefit from graph traversal. Explanatory questions benefit from vector similarity. The router agent ensures each question goes to the optimal source.

Temporal reasoning built-in. The temporal normalizer agent handles date range logic uniformly across all data sources. Once "last 5 seasons" is normalized to a season list, every downstream agent can use that filter without reimplementing temporal logic.

Provenance tracking. The orchestrator logs every step of the reasoning pipeline (temporal normalization, routing decision, agent execution, source retrieval). This enables debugging, anti-hallucination verification, and explainability.

Extensibility. Adding a new agent (e.g., a sentiment analysis agent over fan tweets) requires only implementing the agent interface and registering it with the orchestrator. No changes to existing agents are needed.
