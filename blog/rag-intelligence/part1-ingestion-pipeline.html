Building an Enterprise RAG System from Scratch

RAG Document Intelligence Series · Part 1 of 3

This is Part 1 of a 3-part series where I build a production-ready RAG document intelligence system and show every component, every decision, and every real input/output along the way. Part 2 covers the query pipeline. Part 3 covers architectural decisions and lessons learned.

Everyone talks about RAG. Few show you what it actually looks like inside — the parsing edge cases, the chunking strategy tradeoffs, the moment you realize your 3072-dimensional vectors need a home. By the end of this series, you'll understand not just what RAG is, but why each piece exists and what happens when you get it wrong.

What We're Building

A system where users can:

  1. Upload documents (PDF, DOCX, TXT)
  2. Ask natural-language questions
  3. Get answers grounded in the documents with source citations like [Doc: company_policy.txt, Page: 1]

The stack:

  • FastAPI backend with async endpoints
  • LlamaIndex for sentence-window chunking
  • Azure OpenAI for embeddings (text-embedding-3-large) and generation
  • Qdrant for vector storage and search
  • Streamlit for the frontend
  • Docker Compose to tie it all together

Let's build it piece by piece.

Step 1: Parsing Documents

The parser is the unsexy foundation that everything else depends on. Get it wrong, and your chunks contain garbage, your embeddings encode garbage, and your LLM confidently hallucinates from garbage.

I support three formats with dedicated libraries:

import fitz  # PyMuPDF
from docx import Document as DocxDocument

PARSERS = {
    ".pdf": parse_pdf,    # PyMuPDF - page-level text extraction
    ".docx": parse_docx,  # python-docx - paragraph joining
    ".txt": parse_txt,    # Built-in - just read_text()
}

Each parser returns a list of ParsedPage objects:

@dataclass
class ParsedPage:
    text: str
    page_number: int
    filename: str

Why page-level granularity? Because citations need it. When the LLM says [Doc: report.pdf, Page: 7], the user can go verify. If you flatten everything into one blob of text, you lose this traceability.

Real output from parsing our test document (a 59-line employee handbook):

[ParsedPage(
    text="Acme Corp Employee Handbook - 2025 Edition\n\nChapter 1...",
    page_number=1,
    filename="company_policy.txt"
)]
# 1 page, 1,770 bytes - parsed in 0.1ms

Step 2: Chunking — Where Most RAG Systems Go Wrong

This is where it gets interesting. Chunking strategy is the single biggest lever you have over retrieval quality, and most tutorials gloss over it with RecursiveCharacterTextSplitter(chunk_size=500).

The problem with fixed-size chunking:

"...submitted within 30 days of the expense being incurred. Receipts are"
"required for all expenses over $25. Reimbursements are processed within"

That first chunk ends mid-thought. If someone asks "Do I need receipts for a $20 expense?", the relevant information is split across two chunks. The retriever might only find one of them.

Sentence-Window Chunking

I use LlamaIndex's SentenceWindowNodeParser, which does something clever: it stores two versions of each chunk.

from llama_index.core.node_parser import SentenceWindowNodeParser

node_parser = SentenceWindowNodeParser.from_defaults(
    window_size=3,  # 3 sentences on each side
    window_metadata_key="window",
    original_text_metadata_key="original_text",
)

For each sentence in the document, it produces:

Field What it stores What it's used for
.text The core sentence Shown to the LLM as context
.window Core + 3 sentences before + 3 after Embedded into the vector

Why? The embedding captures semantic meaning from the wider context (the window), so retrieval is more accurate. But the text shown to the LLM is focused and concise, so you don't waste tokens on surrounding fluff.

Real output — 15 chunks from a single page:

Chunk(
    text="Acme Corp allows all full-time employees to work remotely up to 3 days per week.",
    window="Acme Corp Employee Handbook... Remote work days must be scheduled...",
    filename="company_policy.txt",
    page_number=1,
    chunk_index=0
)

The core text is one sentence. The window is seven sentences. The embedding sees the full picture; the LLM gets the precise answer.

Latency: 8.2ms for 15 chunks. Chunking is never the bottleneck.

Step 3: Embedding and Indexing

Each chunk's window text gets embedded into a 3072-dimensional vector using Azure OpenAI's text-embedding-3-large model, then stored in Qdrant.

EMBEDDING_DIMENSION = 3072  # text-embedding-3-large

def ingest_chunks(chunks: list[Chunk]) -> int:
    # Embed the WINDOW text, not the core text
    window_texts = [chunk.window for chunk in chunks]

    for i in range(0, len(window_texts), batch_size):
        batch_texts = window_texts[i : i + batch_size]
        embeddings = embed_texts(batch_texts, openai_client)

        points = [
            PointStruct(
                id=str(uuid.uuid4()),
                vector=embedding,
                payload={
                    "text": chunk.text,        # Core sentence
                    "window": chunk.window,    # Full context
                    "filename": chunk.filename,
                    "page_number": chunk.page_number,
                    "chunk_index": chunk.chunk_index,
                },
            )
            for embedding, chunk in zip(embeddings, batch_chunks)
        ]
        qdrant_client.upsert(collection_name=collection, points=points)

A few things to notice:

  1. We embed the window, but store the core text in the payload. This is the key insight of sentence-window chunking. The vector represents the broad context; the payload contains the focused answer.
  2. The payload carries everything needed for citations — filename, page number, chunk index, and the original text.
  3. Batch size of 100. Azure OpenAI's embedding endpoint handles batches efficiently. For our 15-chunk document, it's a single API call.

Real numbers:

  • 15 chunks embedded in a single batch
  • Each vector: 3072 floats (12 KB)
  • API latency: 387ms
  • Qdrant upsert: ~10ms

The Ingest Endpoint

All three steps are orchestrated by a single FastAPI endpoint:

@router.post("/", response_model=IngestResponse)
async def ingest_document(file: UploadFile = File(...)):
    with trace_ingestion(file.filename) as trace:  # LangFuse tracing
        pages = parse_document(tmp_path)             # Step 1: Parse
        chunks = chunk_pages(pages)                   # Step 2: Chunk
        indexed = ingest_chunks(chunks)               # Step 3: Embed + Index

    return IngestResponse(
        filename=file.filename,
        pages_parsed=len(pages),
        chunks_created=len(chunks),
        chunks_indexed=indexed,
    )

Real response:

{
    "filename": "company_policy.txt",
    "pages_parsed": 1,
    "chunks_created": 15,
    "chunks_indexed": 15
}

Upload a 59-line text file, get 15 semantically-windowed chunks stored as 3072-dimensional vectors in under 400ms. The document is now queryable.
