The Query Pipeline: Hybrid Retrieval, Cross-Encoder Reranking, and Cited Answers

RAG Document Intelligence Series · Part 2 of 3

This is Part 2 of a 3-part series. In Part 1, we parsed documents, split them into sentence-window chunks, embedded them with Azure OpenAI, and stored them in Qdrant. Now comes the hard part: turning a natural-language question into an accurate, cited answer.

This is where most RAG systems fall flat — not because the LLM is bad, but because the retrieval is mediocre. If you feed the LLM the wrong chunks, it will confidently generate a wrong answer.

The query pipeline has three stages, and each one exists because the previous one isn't good enough on its own.

User Question
     |
     v
[ Hybrid Retriever ] --> 20 candidates (dense + BM25 scoring)
     |
     v
[ Cross-Encoder Reranker ] --> 5 candidates (precision-refined)
     |
     v
[ LLM Generator ] --> Answer with [Doc: filename, Page: N] citations

Stage 1: Hybrid Retrieval — Why Dense Search Alone Isn't Enough

The obvious approach: embed the question, find the nearest vectors in Qdrant, done. This is pure dense retrieval, and it works well for semantic queries like "What's the policy on working from home?" matching a chunk about "remote work."

But it fails on precise queries. Ask "What's the minimum internet speed for remote work?" and you need the chunk containing "50 Mbps." Dense embeddings are notoriously bad at encoding specific numbers, acronyms, and technical terms.

That's why I combine dense retrieval (semantic) with BM25 scoring (keyword):

def hybrid_search(question: str, top_n: int = 20, dense_weight: float = 0.7):
    # Phase 1: Dense retrieval from Qdrant
    dense_results = _dense_search(question, qdrant, top_n=top_n)

    # Phase 2: BM25 keyword scoring on the retrieved candidates
    candidates = _bm25_rerank(question, dense_results)

    # Phase 3: Normalize and combine
    for c in candidates:
        norm_dense = (c["dense_score"] - min_dense) / dense_range
        norm_bm25 = (c["bm25_score"] - min_bm25) / bm25_range
        c["score"] = 0.7 * norm_dense + 0.3 * norm_bm25

    candidates.sort(key=lambda c: c["score"], reverse=True)
    return candidates[:top_n]

How Each Phase Works

Dense search embeds the question into the same 3072-dimensional space as the chunks, then finds the 20 nearest vectors by cosine similarity.

BM25 scoring is a classical information retrieval algorithm. It tokenizes both the question and each candidate's text, then scores based on term frequency, inverse document frequency, and document length normalization.

Hybrid fusion normalizes both scores to [0, 1] and combines them:

hybrid_score = 0.7 x normalize(dense_score) + 0.3 x normalize(bm25_score)

The 0.7/0.3 weighting favors semantic understanding but gives keyword matches a meaningful voice.

Why 0.7/0.3?

Dense WeightBM25 WeightObservation
1.00.0Misses exact-term matches ("50 Mbps", "Tier 3")
0.50.5Over-weights keyword matches; semantic quality drops
0.70.3Best balance — semantic-first with keyword boost
0.01.0Pure keyword; misses paraphrased content entirely

Stage 2: Cross-Encoder Reranking — Why 20 Candidates Become 5

Hybrid search gave us 20 candidates sorted by a combined score. But this score is still approximate — the dense embedding encodes query and passage independently (bi-encoder), and BM25 is pure keyword overlap. Neither truly understands whether a passage answers the question.

Enter the cross-encoder. Unlike bi-encoders that encode query and passage separately, a cross-encoder processes the (query, passage) pair jointly through a transformer.

from sentence_transformers import CrossEncoder

_model: CrossEncoder | None = None  # Lazy-loaded singleton

def rerank(question: str, candidates: list[dict], top_k: int = 5):
    model = _get_model()
    pairs = [(question, c["text"]) for c in candidates]
    scores = model.predict(pairs)

    for candidate, score in zip(candidates, scores):
        candidate["rerank_score"] = float(score)

    candidates.sort(key=lambda c: c["rerank_score"], reverse=True)
    return candidates[:top_k]

The Architecture Difference

Bi-Encoder (used in retrieval):
  "How many days remote?" -> [0.12, -0.45, ...]  --+
                                                      +-- cosine = 0.78
  "...remotely up to 3 days per week" -> [0.15, -0.42, ...]  --+
  Fast (one embedding per text) but approximate.

Cross-Encoder (used in reranking):
  ["How many days remote?", "...remotely up to 3 days per week"]
                    |
              Joint transformer encoding
                    |
              Relevance score: 9.14
  Slow (one forward pass per pair) but precise.

The model (ms-marco-MiniLM-L-6-v2, 22M parameters) is small enough to run on CPU but powerful enough to meaningfully reorder the candidates.

The Cold Start Problem

The first time the model loads: 11.1 seconds. Every subsequent call: 1.7 seconds. In production, you'd preload the model during API startup.

Stage 3: LLM Generation — Turning Chunks into Cited Answers

The final stage takes the 5 best chunks and asks the LLM to synthesize an answer with source citations.

The System Prompt

This is the most important prompt in the entire system. It enforces three behaviors:

SYSTEM_PROMPT = """\
You are a precise document Q&A assistant. Answer the user's question using ONLY \
the retrieved context below. For every factual claim, cite the source as \
[Doc: <filename>, Page: <page>]. If the context does not contain enough \
information to answer, say "I don't have enough information in the indexed \
documents to answer this question." Never fabricate information."""
  1. ONLY the retrieved context — prevents hallucination from parametric knowledge
  2. Cite every claim — forces traceability to source documents
  3. Admit ignorance — better to say "I don't know" than fabricate

Real Output

"Employees can work remotely up to 3 days per week
[Doc: company_policy.txt, Page: 1]. However, employees
in their first 90 days (probation period) are required
to work on-site 5 days per week
[Doc: company_policy.txt, Page: 1]."

The model cites both the general policy and the exception. Both citations reference the correct document and page. This is what grounded generation looks like.

Latency: 8,751ms (including ~64 reasoning tokens + ~52 output tokens).

The bottleneck is clear: 77% of warm latency is the LLM call. The retrieval pipeline (embedding + search + BM25 + rerank) takes 2s total.

Streaming: Perceived Latency vs. Actual Latency

The system includes a streaming endpoint that delivers tokens as they arrive via Server-Sent Events:

@router.post("/stream")
async def query_documents_stream(req: QueryRequest):
    async def event_stream():
        candidates = hybrid_search(req.question, top_n=20)
        top_chunks = rerank(req.question, candidates, top_k=req.top_k)

        async for token in generate_answer_stream(req.question, top_chunks):
            yield f"data: {token}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")

With streaming, the user sees the first word in ~2s (retrieval + reranking) instead of waiting 11s for the complete answer. Total time is the same, but perceived latency drops dramatically.

The Observability Layer

Every query is traced via LangFuse, and the trace_id is returned to the frontend. Users can give thumbs up/down on answers, linked directly to the trace that produced them. This closes the loop for evaluation and improvement.
