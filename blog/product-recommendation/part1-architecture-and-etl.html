Architecture & ETL Pipeline — Building a Semantic Product Recommendation Engine

Semantic Product Recommendation Series · Part 1 of 3

Part 1 of 3: How we designed a multimodal recommendation system that ingests 100 real Amazon products, generates dual-vector embeddings, and indexes them for similarity search in under 9 minutes. Part 2 covers the 4 recommendation strategies. Part 3 covers the frontend, deployment, and lessons learned.

The Problem

Product recommendations are everywhere — Amazon, Netflix, Spotify — but building one from scratch exposes some genuinely interesting engineering problems:

  • How do you represent a product as a searchable vector when it has both text descriptions and images?
  • How do you combine visual and semantic similarity in a single search query?
  • How do you personalize results for a user who has only interacted with 3 products?

This series walks through a production-grade answer to all three, built entirely on Azure AI services. No ML model training. No GPU infrastructure. Just clever orchestration of managed services.

The Key Design Decision: Text-Unified Multimodal Embeddings

This is the architectural decision that makes everything else work. Rather than using separate embedding models for text and images (which would produce incomparable vector spaces), we convert everything to text first, then embed with the same model.

The text path:

Product description
  → GPT-4o-mini enrichment
  → "The 245XL Black Ink Compatible cartridges are designed for
     a wide range of Canon printers..."
  → text-embedding-3-small
  → text_embedding [1536 floats]

The image path:

Product image (Amazon CDN URL)
  → Azure AI Vision
  → caption: "a black ink cartridge with a blue box"
  → tags: ["text", "font", "screenshot", "logo"]
  → combined: "a black ink cartridge with a blue box. Tags: text, font, screenshot, logo"
  → text-embedding-3-small (same model!)
  → image_text_embedding [1536 floats]

Both vectors live in the same 1536-dimensional space. This means we can search across text and image fields simultaneously — and weight them differently depending on the strategy. For similar products, we use 70% text + 30% image. For visual search, we use 100% image. Same index, different queries.

The trade-off: A CLIP-style joint embedding would preserve more visual features. But this approach is simpler, uses only managed Azure services, and produces surprisingly good results. When we uploaded an image of a grip strength trainer, the top result came back at score 0.9999 — near-perfect match.

The Search Index: Dual-Vector HNSW

Azure AI Search stores both vector fields in a single index with 13 fields total:

fields = [
    SimpleField(name="id", type=String, key=True, filterable=True),
    SearchableField(name="name", type=String, analyzer_name="en.microsoft"),
    SearchableField(name="description", type=String, analyzer_name="en.microsoft"),
    SearchableField(name="enriched_description", type=String, analyzer_name="en.microsoft"),
    SimpleField(name="category", type=String, filterable=True, facetable=True),
    SimpleField(name="price", type=Double, filterable=True, sortable=True),
    SearchField(
        name="text_embedding",
        type=Collection(Single),
        searchable=True, hidden=False,
        vector_search_dimensions=1536,
        vector_search_profile_name="default-vector-profile",
    ),
    SearchField(
        name="image_text_embedding",
        type=Collection(Single),
        searchable=True, hidden=False,
        vector_search_dimensions=1536,
        vector_search_profile_name="default-vector-profile",
    ),
]

The hidden=False on the vector fields is critical — without it, you can't retrieve the vectors back from the index, which you need for the similar-products strategy.

Live index stats: 100 documents, 4.0 MB total storage, 3.6 MB vector index (90% of storage is vectors), HNSW algorithm with cosine distance.

The ETL Pipeline: 4 Steps, 8 Minutes

The pipeline runs as a FastAPI background task. The caller gets back a run_id immediately and can poll for status. Each step updates its progress in Cosmos DB.

Step 1: Feature Extraction (GPT-4o-mini)

Raw Amazon product descriptions are messy. GPT-4o-mini cleans them up:

EXTRACTION_PROMPT = """\
You are a product feature extraction assistant. Given a product name, description,
and category, extract structured features.

Return a JSON object with these exact keys:
- "tags": list of 5-8 relevant tags
- "key_features": list of 3-5 key product features
- "enriched_description": a 2-3 sentence enriched description optimized for semantic search
"""

The enriched description is what gets embedded — it's what semantic search actually sees. The raw Amazon description with its spec lists and model numbers would produce a much worse embedding.

Step 2: Image Analysis (Azure AI Vision)

result = client.analyze_from_url(
    image_url=image_url,
    visual_features=[VisualFeatures.CAPTION, VisualFeatures.TAGS],
)
caption = result.caption.text
tags = [tag.name for tag in result.tags.list if tag.confidence > 0.5]
image_text = f"{caption}. Tags: {', '.join(tags)}"

For products without valid images, the pipeline synthesizes image text from the product name, category, and tags — so every product gets both vector fields populated.

Step 3: Embedding Generation (text-embedding-3-small)

Both the enriched description and the image text are embedded in batches of 50. 100 products = 4 API calls (2 for text, 2 for image-text). This is the fastest step at ~30 seconds.

Step 4: Indexing (AI Search + Cosmos DB)

Products are split across two stores: Azure AI Search (full documents with vectors for similarity search) and Cosmos DB (metadata without vectors for CRUD, stats, interaction tracking).

Real Data: HuggingFace Amazon Products

Rather than synthetic data, we built a download script that pulls real Amazon products from HuggingFace datasets across 5 categories: Electronics, Clothing, Home & Kitchen, Sports, and Books. Quality filters ensure usable data: minimum title/description lengths, valid prices, no spammy listings. The final catalog has 100 products (20 per category) with real Amazon CDN image URLs.
