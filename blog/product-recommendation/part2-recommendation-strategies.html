Four Recommendation Strategies — Dual-Vector, Hybrid, Visual, and Personalized

Semantic Product Recommendation Series · Part 2 of 3

This is Part 2 of a 3-part series. In Part 1, we built the ETL pipeline. Now let's put 100 indexed products to work with 4 recommendation strategies. Part 3 covers the frontend, deployment, and lessons learned.

Strategy 1: Similar Products (Dual-Vector Search)

Use case: User is viewing a product and wants to see related items.

Retrieve the source product's text and image embeddings from the index, then search both vector fields simultaneously with different weights:

def get_similar_products(product_id: str, top_k: int = 8) -> list[dict]:
    # 1. Look up source product's vectors
    doc = search_index.get_document_by_id(product_id)
    text_vec = doc["text_embedding"]         # 1536 floats
    image_vec = doc["image_text_embedding"]  # 1536 floats

    # 2. Dual-vector search with weighted scoring
    results = search_index.search_by_dual_vector(
        text_vector=text_vec,
        image_vector=image_vec,
        top_k=top_k + 1,
        exclude_ids=[product_id],
    )
    return results[:top_k]

Under the hood, Azure AI Search receives two VectorizedQuery objects:

text_vq = VectorizedQuery(
    vector=text_vec,
    k_nearest_neighbors=top_k,
    fields="text_embedding",
    weight=0.7,                  # Text similarity matters more
)
image_vq = VectorizedQuery(
    vector=image_vec,
    k_nearest_neighbors=top_k,
    fields="image_text_embedding",
    weight=0.3,                  # Image similarity is supplementary
)

The 0.7/0.3 weighting is intentional. Product descriptions contain richer semantic information (features, use cases, specifications) than image captions ("a black ink cartridge with a blue box"). But the image signal catches cross-category visual similarities that text misses.

Strategy 2: Text Search (Hybrid Vector + BM25)

Use case: User types a natural language query like "wireless noise-cancelling headphones."

The query is both embedded (for semantic search) and passed as raw text (for BM25 keyword matching). Azure AI Search fuses both scores via Reciprocal Rank Fusion (RRF).

def search_by_text(query: str, top_k: int = 10, category: str | None = None):
    query_vector = embedder.embed_single(query)

    return search_index.search_by_text_vector(
        query_vector=query_vector,  # Semantic similarity
        top_k=top_k,
        category=category,          # Optional filter
        search_text=query,           # BM25 keyword matching
    )

Why Hybrid Instead of Pure Vector?

Vector search captures semantic meaning — "wireless headphones" matches "Bluetooth audio device" even with zero keyword overlap. But it can miss exact matches that BM25 nails — "Canon 245XL" should rank a document containing "Canon 245XL" very highly, even if the vector distance isn't the smallest.

Hybrid gives you both:

  • Vector search handles "running shoes for flat feet" → matches "supportive athletic footwear for overpronation"
  • BM25 handles "Canon 245XL" → matches the exact product model

Strategy 3: Visual Search (Image Upload)

Use case: User uploads a photo of a product they want to find.

The uploaded image goes through Azure AI Vision for captioning, the caption is embedded, and the resulting vector is searched against the image_text_embedding field.

def search_by_image(image_data: bytes, top_k: int = 8) -> list[dict]:
    # 1. AI Vision analyzes the uploaded image
    analysis = image_analyzer.analyze_image_from_bytes(image_data)
    image_text = analysis["image_text"]

    # 2. Embed the image description
    image_vector = embedder.embed_single(image_text)

    # 3. Search the image_text_embedding field
    return search_index.search_by_image_vector(
        image_vector=image_vector,
        top_k=top_k,
    )

Strategy 4: Personalized Recommendations

Use case: Returning user with interaction history wants personalized product suggestions.

Instead of a traditional user-item matrix, we compute a weighted preference vector directly from the user's interacted products' embeddings, then search for similar products, excluding items they've already seen.

Interaction Weights

INTERACTION_WEIGHTS = {
    "view": 1.0,       # Passive interest
    "click": 2.0,      # Active interest
    "add_to_cart": 3.0, # Purchase intent
    "purchase": 5.0,    # Strongest signal
}

The Algorithm

def get_personalized_recommendations(user_id: str, top_k: int = 10):
    # 1. Fetch interaction history from Cosmos DB
    interactions = cosmos_db.get_user_interactions(user_id, limit=50)
    if not interactions:
        return _get_popular_products(top_k)  # Cold-start fallback

    # 2. Collect product embeddings with weights
    product_vectors = []
    seen_product_ids = set()
    for interaction in interactions:
        pid = interaction["product_id"]
        seen_product_ids.add(pid)
        weight = INTERACTION_WEIGHTS[interaction["interaction_type"]]
        doc = search_index.get_document_by_id(pid)
        if doc and doc.get("text_embedding"):
            product_vectors.append((doc["text_embedding"], weight))

    # 3. Compute weighted average preference vector
    total_weight = sum(w for _, w in product_vectors)
    preference_vector = [0.0] * 1536
    for vec, weight in product_vectors:
        for i in range(1536):
            preference_vector[i] += vec[i] * weight
    preference_vector = [v / total_weight for v in preference_vector]

    # 4. Search, excluding already-seen products
    results = search_index.search_by_text_vector(
        query_vector=preference_vector,
        top_k=top_k + len(seen_product_ids),
    )
    return [r for r in results if r["id"] not in seen_product_ids][:top_k]

How Azure AI Search Handles Dual-Vector Queries

When we send two VectorizedQuery objects with different weights:

  1. AI Search runs HNSW search on text_embedding → top-K candidates with text scores
  2. AI Search runs HNSW search on image_text_embedding → top-K candidates with image scores
  3. Scores are combined: final_score = 0.7 × text_score + 0.3 × image_score
  4. Results are re-sorted by final score

This happens server-side in a single API call — we don't need to make two separate search requests and fuse them ourselves.
