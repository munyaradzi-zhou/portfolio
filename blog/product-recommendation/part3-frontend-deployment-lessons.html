Deployment: Docker Compose

The system deploys as 2 services (no self-hosted databases — all data services are Azure-managed):

services:
  api:
    build:
      context: .
      dockerfile: Dockerfile.api
    ports:
      - "8006:8006"
    env_file: .env
    healthcheck:
      test: ["CMD", "python", "-c",
             "import urllib.request; urllib.request.urlopen('http://localhost:8006/health')"]
      interval: 15s
      timeout: 10s
      retries: 3
      start_period: 30s

  frontend:
    build:
      context: .
      dockerfile: Dockerfile.frontend
    ports:
      - "8508:8508"
    environment:
      - API_BASE=http://api:8006
    depends_on:
      api:
        condition: service_healthy

The depends_on: condition: service_healthy ensures the frontend doesn't start until the API's health check passes. The API's lifespan handler creates the AI Search index and Cosmos DB containers on startup.

Startup Sequence

1. API container starts
2. Lifespan handler runs:
   a. ensure_index()    → Creates/updates AI Search index (13 fields, HNSW)
   b. ensure_cosmos_db() → Creates database + 3 containers if they don't exist
3. Health check passes (/health returns {"status": "ok"})
4. Frontend container starts
5. Frontend connects to http://api:8006

The Data Model: Cosmos DB

Three containers, all with /id partition key:

  • products (100 documents) — Metadata without vectors. Vectors are only in AI Search. This separation keeps Cosmos DB costs down.
  • interactions — User interaction events (view, click, add_to_cart, purchase) with timestamps
  • pipeline-runs — ETL status tracking with step-by-step progress

Why /id Partition Key?

Using a mutable field like /category as partition key creates problems when documents change category during processing. /id ensures each document has a stable, unique partition. The trade-off is that cross-partition queries require enable_cross_partition_query=True, but at 100 documents, this is negligible.

Lessons Learned

1. Vector Fields Must Be hidden=False. When you create a SearchField with searchable=True and vector_search_dimensions, the default is hidden=True. This means the vector is stored and searchable but cannot be retrieved via get_document(). This broke our similar-products strategy, which needs to look up a product's vectors before searching for neighbors.

2. You Can't Add filterable to an Existing Key Field. Azure AI Search doesn't let you modify the filterable property of an existing key field. We created id without filterable=True, then discovered we needed it for the exclude-ID filter. The workaround: fetch top_k + 1 results and filter in Python. For fresh deployments, ensure_index() now creates id with filterable=True.

3. Real Amazon Data Needs Aggressive Filtering. The HuggingFace Amazon datasets contain noise: "Sold Out" titles, spec-table descriptions, Unicode brackets in tags, string prices. Our download script applies quality filters for minimum lengths, valid prices, and spam keywords.

4. Singleton Clients Save Connection Overhead. Every Azure service client uses the same singleton pattern — a global _client variable initialized on first call. This appears in all 7 service files.

5. Batching Embedding Calls Matters. Step 3 processes 200 texts in 4 API calls (batch size 50). Without batching, this would be 200 separate calls. The difference: ~30 seconds vs ~5+ minutes.

6. CORS Must Be Configured at Module Level. Adding CORS middleware inside FastAPI's lifespan handler causes RuntimeError: Cannot add middleware after an application has started. Configure CORS at module level using os.environ.get().
