The Intelligence Layer: GenAI, Search, and Keeping It All Alive

NovaMart Series · Part 3 of 3

This is Part 3 of a 3-part series on building NovaMart. Part 2 covered data engineering and traditional ML. Now we get to the frontier — the skills that are in the highest demand right now and barely exist in any curriculum at all.

The Skills That Don't Exist in Classrooms Yet

If the data layer is the foundation that every ML engineer needs but college doesn't teach, the intelligence layer is the frontier — the skills that are in the highest demand right now and barely exist in any curriculum at all.

Vector embeddings. Semantic search. Retrieval-Augmented Generation. LLM integration. Streaming responses. Production monitoring. Drift detection.

These aren't tomorrow's skills. They're today's. Companies are hiring for them aggressively, and the supply of people who've actually built these systems is tiny. Most candidates can explain what RAG is. Far fewer have built a RAG pipeline that queries a real product catalog and streams answers to a real user interface.

This is the part of NovaMart that bridges the widest gap between classroom and industry.

Turning Products Into Math

Before NovaMart could search intelligently, I needed to solve a fundamental problem: computers don't understand meaning. They match strings. "Waterproof hiking jacket" and "rain-resistant outdoor shell" are completely different strings that mean nearly the same thing. Traditional keyword search treats them as unrelated.

The solution is embeddings — one of the most important concepts in modern AI, and one that most ML courses mention in passing but never ask you to actually implement in a real system.

Here's how it works. You take a product's text — its name, description, category — and feed it through a language model (I used Azure OpenAI's text-embedding-3-small). What comes back is a vector: a list of 1,536 numbers that captures the semantic meaning of that product. Products with similar meanings end up close together in this 1,536-dimensional vector space, even if they use completely different words.

I generated embeddings for all 32,951 products in the catalog. Each product now has its mathematical fingerprint stored in PostgreSQL alongside its name, price, and description, using the pgvector extension for efficient vector operations.

This is the same technology stack that companies like Shopify, Instacart, and dozens of startups use for production AI search. Learning to generate, store, and query embeddings at scale is one of the most immediately marketable skills in the current job market.

Semantic Search

With embeddings in place, search becomes a similarity problem instead of a keyword-matching problem. Here's what happens when a user types a query:

  1. The query gets converted into its own 1,536-dimensional embedding
  2. PostgreSQL (via pgvector) computes the cosine similarity between the query embedding and all 32,951 product embeddings
  3. The most semantically similar products come back, ranked by relevance

The results are dramatically better than keyword search. "Birthday gift for a teenage girl who likes art" returns sketchbook sets, colored pencil collections, and art supply kits — even though none of those products contain the words "birthday," "teenage," or "girl." The system matched meaning, not words.

"Comfortable shoes for long runs on pavement" returns running shoes with cushioned soles, not hiking boots and shark slippers. "Something to keep my kid entertained on a flight" returns activity sets and portable games, not aviation equipment.

Building this taught me several things that no course covered:

Embedding generation at scale is a pipeline problem. Generating 32,951 embeddings means managing API rate limits, handling failures gracefully, batching requests, and tracking coverage. I built a pipeline that tracks which products have embeddings and which don't, with coverage metrics visible on the admin dashboard.

Vector indexing matters. On a small dataset, brute-force cosine similarity is fine. At scale, you need approximate nearest neighbor indexes (IVFFlat, HNSW). Understanding the tradeoffs between accuracy and speed is a production concern that doesn't come up in tutorials.

Fallbacks are engineering. Not every product has an embedding (coverage isn't 100%). The system falls back to category-based matching when embeddings are missing. Designing graceful degradation paths is a production skill.

The RAG Chatbot

RAG — Retrieval-Augmented Generation — is the most in-demand AI architecture right now. Every company building LLM-powered features uses some version of it. And the gap between "I understand how RAG works" and "I built a RAG system that queries real data and streams responses to a real frontend" is exactly the kind of gap that separates candidates.

NovaMart's chatbot implements the full RAG pipeline:

Step 1: Embed the question. When a user types "I need headphones that won't fall out during a workout," the system converts the question into an embedding using the same model used for products.

Step 2: Retrieve relevant products. That embedding gets searched against the product catalog via pgvector. The top 5 most relevant products are retrieved, with their descriptions, prices, ratings, and reviews.

Step 3: Build context. The retrieved products are assembled into a structured context document — the "retrieval" part of RAG. This is what grounds the LLM's response in real data instead of letting it hallucinate.

Step 4: Generate a response. The context and the user's question are sent to Azure OpenAI's GPT-4o-mini with a carefully crafted system prompt. The model generates a natural-language response that references specific products, explains why they're relevant, and gives genuinely useful shopping advice.

Step 5: Stream it. The response streams back to the frontend via Server-Sent Events (SSE), word by word. The user sees the chatbot "typing" in real time, which makes the interaction feel responsive instead of staring at a loading spinner.

The entire round trip takes a couple of seconds. The chatbot remembers conversation context within a session, so follow-up questions work naturally. "Show me running shoes." "Something cheaper." "Does that one have good reviews?" Each message builds on the last.

Building this taught me skills that are immediately relevant to any company shipping LLM features:

Prompt engineering is real engineering. The system prompt matters enormously. Too permissive and the model hallucinates products that don't exist. Too restrictive and the responses feel robotic. Finding the right balance through iteration is a craft.

Streaming changes everything. SSE streaming is how every production chatbot works (ChatGPT, Copilot, you name it). Implementing it end-to-end — from the API layer through to the React frontend — is a full-stack skill that crosses the boundary between ML and software engineering.

Context window management is a constraint. With 5 retrieved products, their descriptions, and conversation history, token counts add up fast. Learning to manage context windows efficiently — what to include, what to truncate, when to summarize — is a practical LLM engineering skill.

Cost awareness matters. I chose GPT-4o-mini deliberately. It's fast, capable, and costs a fraction of what GPT-4o does. Making architecture decisions that balance capability against cost is how real engineering teams operate. The admin dashboard tracks token consumption and LLM costs so that spending stays visible.

Similar Products

Every product page has a "You might also like" section. In most stores, this is either random or crudely correlated ("you looked at a laptop, here are more laptops"). NovaMart's version uses the same embedding infrastructure as the search.

Since every product already has a 1,536-dimensional embedding, finding similar products is a nearest-neighbor lookup: "what's closest to this product in vector space?"

The results capture thematic relationships that go beyond simple category matching. A camping tent surfaces sleeping bags, camp stoves, and hiking boots. A recipe book surfaces kitchen utensils and ingredient organizers. The embeddings encode semantic relationships that would be impossible to capture with hand-written rules.

This is a small feature, but it exercises an important skill: reusing ML infrastructure across multiple product features. The embedding generation pipeline, the pgvector storage, and the similarity search logic are shared between semantic search, the chatbot's retrieval step, and the similar products feature. Building shared ML infrastructure that serves multiple downstream use cases is how ML platforms work at scale.

Production Monitoring

Here's a truth about ML in production that no college course prepares you for: training a model is the easy part. Keeping it working over time is the hard part.

Models are trained on historical data. When the real world drifts away from that history, predictions degrade. A recommendation engine trained on summer shopping patterns gives irrelevant suggestions in December. A demand forecaster trained before a trend goes viral underestimates demand by orders of magnitude.

This is called data drift, and it's how ML systems silently fail. The model doesn't crash. It doesn't throw an error. It just gets worse, so gradually that nobody notices until customers complain or revenue drops.

I built a monitoring system for NovaMart that watches for this:

Statistical drift detection. The system runs Kolmogorov-Smirnov tests on numerical features and chi-squared tests on categorical ones, comparing current data distributions against training data distributions. When distributions diverge beyond a threshold, the system flags it with severity levels.

Model performance tracking. Each model's key metrics (precision, recall, NDCG for recommendations; MAPE, RMSE for forecasting; silhouette score for segmentation) are tracked and displayed on the admin dashboard. Degradation over time is visible at a glance.

Embedding coverage monitoring. What percentage of products have embeddings, broken down by category? If a new batch of products gets loaded without embeddings, the coverage drops and the admin dashboard shows it immediately. This prevents the semantic search and chatbot from silently degrading as new products are added.

System health metrics. Database size, record counts, API response times. The operational metrics that keep a system running.

OpenTelemetry instrumentation. Every API request is traced end-to-end — how long it took, what services it touched, where time was spent. If semantic search slows down, the traces show whether the bottleneck is the embedding generation, the vector query, or the API logic.

The admin dashboard at /admin shows all of this: sales trends, customer segment distributions, model performance cards, drift summaries with color-coded severity (green/yellow/red), and embedding coverage stats.

None of this is visible to the shopper. They just see a store that works. But behind the curtain, there's a monitoring system that ensures "works" stays true over time. This is MLOps — and it's the skill set that companies are most desperate to hire for, because almost nobody learns it in school.

The Uncomfortable Truth — And What To Do About It

The AI/ML industry has a chicken-and-egg problem. Entry-level positions require practical experience. Practical experience requires a position — or, traditionally, an internship at a company that uses these tools in production.

But that's not the only path. The tools are all open source or have free tiers. The data is publicly available. The documentation is excellent. The only thing missing is a reason to use them all together — a project with enough complexity to demand real engineering, not just toy exercises.

E-commerce gave me that reason. It naturally requires recommendation engines, search systems, data pipelines, forecasting, segmentation, LLM integration, and production monitoring. It forced me to make real architectural decisions, handle real data quality issues, and build systems that work end-to-end.

If you're in the same position I was — strong on theory, short on practical ML engineering experience — the approach isn't complicated:

  1. Pick a domain that naturally requires multiple ML skills. E-commerce, healthcare analytics, content platforms, financial services — anything with real complexity.
  2. Use real data. Kaggle, government open data, public APIs. The messier, the better.
  3. Build the whole system, not just the model. Pipeline, warehouse, training, serving, monitoring. The model is maybe 20% of the work.
  4. Use industry tools. dbt, MLflow, Prefect, pgvector, FastAPI. These are what companies use. Learning them on a real project is infinitely more valuable than reading their docs.
  5. Track and document everything. Experiment tracking, model cards, honest metric reporting. This demonstrates engineering maturity.
  6. Monitor in production. Even if "production" is your laptop. Drift detection and performance tracking show that you understand the ML lifecycle beyond training.

The skills gap is real, but it's not insurmountable. You don't need an internship at a FAANG company to learn production ML engineering. You need a project ambitious enough to demand it.

NovaMart was mine. Build yours.
