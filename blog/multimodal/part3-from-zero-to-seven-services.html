Three layers, three responsibilities: Services wrap each Azure SDK, Routes wire those functions to HTTP endpoints, and the Frontend calls the endpoints and renders results. Each layer only knows about the layer below it.

Pattern 1: The Config File

With 7 Azure services, you have 15 configuration values. Scattering these across different files is a recipe for chaos. Instead, we put them all in one place:

from pydantic_settings import BaseSettings
from functools import lru_cache

class Settings(BaseSettings):
    # Azure AI Vision
    azure_vision_endpoint: str
    azure_vision_key: str

    # Azure AI Speech (region, NOT endpoint)
    azure_speech_key: str
    azure_speech_region: str

    # Azure AI Language
    azure_language_endpoint: str
    azure_language_key: str

    # ... and so on for all 7 services

    model_config = {"env_file": ".env", "extra": "ignore"}

@lru_cache
def get_settings() -> Settings:
    return Settings()

This does three powerful things:

  1. Validation at startup. If any required key is missing, the app crashes immediately with a clear error — not 10 minutes later when someone tries to upload an image.
  2. Single import. Every service file does from config import get_settings and gets access to everything.
  3. Caching. The @lru_cache decorator means the settings are loaded once and reused.

Pattern 2: The Singleton Service Client

Each Azure service needs a client object. Creating a new client for every request is wasteful. So we use a simple singleton pattern:

_client = None  # Module-level variable

def get_vision_client():
    global _client
    if _client is None:
        settings = get_settings()
        _client = ImageAnalysisClient(
            endpoint=settings.azure_vision_endpoint,
            credential=AzureKeyCredential(settings.azure_vision_key),
        )
    return _client

First call: creates the client. Every call after that: returns the same client. We used this pattern for all 7 services. The only variation is the Speech SDK, which creates a SpeechConfig each time because recognizer/synthesizer objects are inherently single-use.

Pattern 3: UUID-Prefixed File Names

When you let users upload files, name collisions are inevitable. Our solution: prefix every filename with 8 random hexadecimal characters.

import uuid

def upload_file(filename, content, content_type):
    blob_name = f"{uuid.uuid4().hex[:8]}_{filename}"
    # "image.jpg" becomes "f67c8117_image.jpg"

The original filename is preserved (useful for display), but the stored name is unique. The chance of a collision with 8 hex characters is about 1 in 4 billion.

Building the Service Layer: Seven Wrappers

Vision Client — 5 Features in One Call

The Azure Vision SDK lets you request multiple visual features in a single API call:

result = client.analyze(
    image_data=image_bytes,
    visual_features=[
        VisualFeatures.CAPTION,
        VisualFeatures.DENSE_CAPTIONS,
        VisualFeatures.TAGS,
        VisualFeatures.OBJECTS,
        VisualFeatures.READ,       # OCR
    ],
)

One API call, five types of analysis. Both faster and cheaper than making separate calls.

Language Client — 5 Features, 5 Calls

The Language SDK is different. Each NLP feature is a separate method call:

lang_results = client.detect_language(documents=[text])
sent_results = client.analyze_sentiment(documents=[text])
ner_results  = client.recognize_entities(documents=[text])
pii_results  = client.recognize_pii_entities(documents=[text])
kp_results   = client.extract_key_phrases(documents=[text])

That's 5 API calls per analysis. On the free tier (5,000 transactions/month), this means about 1,000 full analyses per month.

Speech Client — The Most Complex Service

Speech-to-text with continuous recognition is the most code-intensive wrapper:

def transcribe_audio(audio_bytes, filename):
    # 1. Write bytes to a temp file
    with tempfile.NamedTemporaryFile(suffix=".wav", delete=False) as tmp:
        tmp.write(audio_bytes)
        tmp_path = tmp.name

    # 2. Set up recognition with callbacks
    recognizer = SpeechRecognizer(speech_config, AudioConfig(filename=tmp_path))

    all_results = []
    done = threading.Event()

    def on_recognized(evt):
        all_results.append(evt.result.text)

    def on_session_stopped(evt):
        done.set()

    recognizer.recognized.connect(on_recognized)
    recognizer.session_stopped.connect(on_session_stopped)

    # 3. Start and wait
    recognizer.start_continuous_recognition()
    done.wait(timeout=120)
    recognizer.stop_continuous_recognition()

    # 4. Clean up
    os.unlink(tmp_path)

    return {"text": " ".join(all_results), "segments": all_results}

The key insight: continuous recognition is event-driven. The on_recognized callback fires each time a segment of speech is transcribed. The on_session_stopped callback fires when the entire file has been processed. This is fundamentally different from the other services, which are simple request-response calls.

The API Layer: Routes That Orchestrate

Each route file is about 40–70 lines. They don't contain business logic — they orchestrate the service layer:

@router.post("/analyze")
async def analyze_image(image: UploadFile = File(...)):
    content = await image.read()                    # 1. Read input
    safety = content_safety.check_image(content)    # 2. Safety check
    blob_url = blob_storage.upload_file(...)        # 3. Store the file
    result = vision_client.analyze_image(content)   # 4. Do the analysis
    cosmos_db.save_analysis({...})                   # 5. Save to database
    return result                                   # 6. Return to user

Read, check, store, analyze, save, return. Every route follows this rhythm.

The Docker Setup: Two Containers, One Network

services:
  api:
    build:
      dockerfile: Dockerfile.api
    ports:
      - "8005:8005"
    env_file: .env
    healthcheck:
      test: ["CMD", "python", "-c",
             "import urllib.request; urllib.request.urlopen('http://localhost:8005/health')"]

  frontend:
    build:
      dockerfile: Dockerfile.frontend
    ports:
      - "8507:8507"
    environment:
      - API_BASE=http://api:8005
    depends_on:
      api:
        condition: service_healthy

Two important details:

  1. The frontend waits for the API. The depends_on: condition: service_healthy line means Docker won't start the frontend until the API's health check passes.
  2. The API Dockerfile installs system libraries. The Speech SDK needs libssl-dev (for encrypted connections) and libasound2 (for audio processing). Without these, the container crashes on import with a cryptic shared library error.

Lessons Learned

1. Speech SDK auth is different from everything else. Every other Azure AI service uses Client(endpoint=URL, credential=Key). Speech SDK uses SpeechConfig(subscription=key, region="eastus"). Region is a short string, not a URL. This will waste 30 minutes of your life if you don't know it.

2. Free tiers are generous enough for real development. We ran dozens of tests across all 4 new services and never came close to the free tier limits.

3. Content Safety is remarkably fast. Each safety check takes about 200–500ms. The GPT-4o call at 5–8 seconds is always the bottleneck.

4. Sentiment analysis measures words, not intent. "I overcame terrible obstacles" scores as partly negative because of "terrible obstacles." If your application needs to detect intent rather than tone, you'll need GPT-4o for that.

5. Cosmos DB partition keys should be immutable. Use the document ID (which never changes) as your partition key, not a field that might change during processing.

6. Base64 encoding inflates image size by 33%. A 100 KB image becomes about 133 KB when base64-encoded for GPT-4o. For large images, consider resizing before encoding.

7. Docker containers don't have audio hardware. File-based I/O with temp files is the solution, and it works perfectly.

8. The @lru_cache config pattern prevents repeated file reads. Without caching, every service call would re-read the .env file.

What Would We Do Differently?

Add async calls where possible. Our Vision, Language, and Content Safety calls are synchronous. Making them async with asyncio.gather() would let us run analyses in parallel during the multimodal endpoint, cutting latency by 2–3 seconds.

Add image resizing. Large images (5+ MB) sent to GPT-4o as base64 are expensive in tokens. A preprocessing step that resizes to 1024px max would cut costs with minimal quality loss.

Add response caching. If the same image is analyzed twice, we currently make all the API calls again. A content-hash-based cache would save time and money.

But these are optimizations, not necessities. The system works well as-is, and premature optimization is the root of all evil.

The One-Command Deploy

After all that architecture, deploying the whole thing is one command:

docker compose up --build -d

Ninety seconds later, the API is live on port 8005, the dashboard is live on port 8507, and you're analyzing images, transcribing audio, and detecting PII. Seven Azure services, two containers, zero manual configuration beyond the .env file.

That's the power of good architecture: complexity behind the scenes, simplicity at the surface.
