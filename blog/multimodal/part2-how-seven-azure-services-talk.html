How 7 Azure Services Talk to Each Other

Multimodal AI Assistant Series · Part 2 of 3

This is Part 2 of a 3-part series. In Part 1, we introduced the multimodal AI assistant and what it can do. Now let's talk about how it does it. Part 3 covers the implementation and lessons learned.

The Big Picture: A Restaurant Kitchen

Think of our system like a restaurant kitchen. The customer (user) places an order at the front counter (Streamlit dashboard). The order goes to the kitchen (FastAPI backend), where different chefs (Azure services) each do their part:

  • The Vision Chef examines what's on the plate (analyzes images)
  • The Speech Chef takes verbal orders and reads them back (transcribes and synthesizes audio)
  • The Language Chef reads written notes and highlights the important parts (entity extraction, sentiment)
  • The Safety Inspector checks every ingredient coming in and every dish going out (content moderation)
  • The Head Chef (GPT-4o) combines everyone's work and adds the creative touch (multimodal reasoning)
  • The Filing Clerk saves photos of every dish (Blob Storage)
  • The Bookkeeper records every order in the ledger (Cosmos DB)

No single chef does everything. But together, they produce something none of them could alone.

Tracing a Real Request: "What Color Are the Tree Leaves?"

Let's follow a real request from our live test. A user uploaded a photo of an autumn tree and asked: "What color are the tree leaves?"

STEP 1:  User clicks "Ask GPT-4o" in the Vision tab
         → Browser sends image file + question text to the API

STEP 2:  API receives the request at POST /vision/ask
         → Reads the image bytes (87,432 bytes of JPEG data)
         → Reads the question: "what color are the tree leaves?"

STEP 3:  Content Safety checks the IMAGE
         → Response: {Hate: 0, SelfHarm: 0, Sexual: 0, Violence: 0}
         → is_safe = true ✓

STEP 4:  Content Safety checks the QUESTION TEXT
         → Response: all zeros, is_safe = true ✓

STEP 5:  GPT-4o analyzes the image + question
         → Image bytes converted to base64 string
         → Sent to GPT-4o with both the image and question
         → Answer: "The tree leaves are primarily shades of red, orange,
           and yellow, reflecting the vibrant colors typically seen
           during autumn."

STEP 6:  Content Safety checks GPT-4o's ANSWER
         → Response: all zeros, is_safe = true ✓

STEP 7:  Blob Storage saves the image
         → Filename: "f67c8117_pexels-pixabay-33109.jpg"

STEP 8:  Cosmos DB saves the analysis record

STEP 9:  API returns the response to the browser

STEP 10: Streamlit renders the Q&A card with safety badge

Services involved in this single request: 4 (Content Safety ×3 calls, GPT-4o, Blob Storage, Cosmos DB)

Total time: About 5–10 seconds, most of which is GPT-4o thinking.

Notice that Content Safety was called three times — once for the image, once for the question, and once for the answer. This is "defense in depth." We don't trust that the input is safe just because it looks harmless. And we don't trust that the AI's output is safe just because the input was safe. We check everything, every time.

The Speech Puzzle: No Microphone in a Docker Container

Here's a problem that's easy to overlook: our application runs inside Docker containers. A Docker container is like a lightweight virtual computer — it has a file system, it can run programs, but it doesn't have hardware like a microphone or speakers.

The answer is file-based I/O. Instead of listening through a microphone, the Speech SDK reads from a file. Instead of playing through a speaker, it writes to a file.

Speech-to-Text

  1. User uploads an audio file (say, a 20-second WAV recording)
  2. The API receives the raw bytes
  3. We write those bytes to a temporary file on disk
  4. We tell the Speech SDK: "Read from this file instead of a microphone"
  5. The SDK transcribes the audio
  6. We delete the temp file and return the text

Text-to-Speech

  1. User types some text and picks a voice
  2. We tell the Speech SDK: "Write to this file instead of playing through a speaker"
  3. The SDK generates audio and saves it as a WAV file
  4. We read the bytes, delete the temp file, and send audio back to the browser
  5. The browser plays it using its own audio capabilities

It's a workaround, but it's a clean one. The user never knows the difference.

Why the Speech SDK Is the Odd One Out

Every Azure AI service follows the same authentication pattern — you give it an endpoint URL and an API key:

Vision:         endpoint = "https://my-vision.cognitiveservices.azure.com/"
Language:       endpoint = "https://my-language.cognitiveservices.azure.com/"
Content Safety: endpoint = "https://my-safety.cognitiveservices.azure.com/"
OpenAI:         endpoint = "https://my-openai.openai.azure.com/"

Then there's the Speech SDK:

Speech:         region = "eastus"    ← Wait, what?

No endpoint URL. Just a region string like "eastus" or "westus2." The Speech SDK takes that region, constructs the endpoint URL internally, and connects. This is because the Speech SDK ships as a native C/C++ library (not pure Python like the others), and it has its own way of handling connections.

This trips up almost everyone the first time. You copy the endpoint URL from the Azure portal and try to use it — and nothing works. The fix: use just the region name, not the full URL. We spent more time debugging this one line of configuration than any other part of the project.

How Images Get Sent to GPT-4o

GPT-4o is a text-based API — you send it messages as JSON text. So how do you send it an image?

The answer is base64 encoding. Base64 is a way to represent binary data (like the bytes of a JPEG file) as plain text characters:

  1. You have an image file: 87,432 bytes of JPEG data
  2. You convert it to base64: a text string of about 116,576 characters
  3. You wrap it in a data URL: data:image/jpeg;base64,/9j/4AAQ...
  4. You include that string in the API message alongside your question

The base64 encoding makes the message about 33% larger than the original image, but it means you can send images through a text-only API without any special image upload protocol.

The Database Decision That Saved Us Hours

Azure Cosmos DB requires you to choose a "partition key" — a field in your data that determines how the database organizes documents internally.

In an earlier project, we chose the document type as the partition key. That seemed logical: all invoices go in one partition, all contracts in another. But when a document got re-classified during processing, it effectively needed to move between partitions, which Cosmos DB doesn't allow.

For this project, we chose the document ID as the partition key. Every document gets a unique ID when it's created, and that ID never changes. This is one of those decisions that sounds trivial but saves you from bizarre bugs at 2 AM.

Content Safety: Why We Check Everything Twice

You might wonder why we run Content Safety checks on both inputs and outputs. If the input is safe, won't the output be safe too?

Not necessarily:

  • Safe input, unsafe output: A user uploads a perfectly innocent landscape photo. But due to a quirk in GPT-4o's training, it generates a description that includes inappropriate content.
  • Unsafe input that bypasses text checks: A user uploads an image that contains harmful content. Text-based moderation can't catch this — you need image-specific moderation.

So we check every uploaded image, every piece of user text, and every AI-generated response. If anything fails, we flag it with a red "Flagged" badge instead of a green "Safe" badge.

The Sentiment Surprise

One of our live tests produced an unexpected result. We analyzed a student campaign speech — an upbeat, motivational text about organizing fundraisers and gathering signatures. You'd expect positive sentiment, right?

The Language service scored it as "mixed" — 31% positive, 18% neutral, and 50% negative.

Why? Because the speech contains phrases like "budget cuts," "didn't just complain," "created chaos," and "I didn't just wait." These are negative constructions, even though the intent is positive.

Lesson: Sentiment analysis measures the emotional tone of the words, not the intent of the speaker. A speech that says "we overcame terrible challenges" will score as partly negative because of "terrible challenges," even though the overall message is triumphant.

PII Detection: The Invisible Shield

Our Language service automatically detects PII (Personally Identifiable Information) and can redact it:

Original:
"Good morning, Riverside High! I'm Sarah Martinez, and I'm running for student body president."

Redacted:
"Good morning, **************! I'm **************, and I'm running for ************ *********."

This is incredibly useful for any application that handles user data — customer service logs, healthcare records, or any system that needs to anonymize personal details automatically.
